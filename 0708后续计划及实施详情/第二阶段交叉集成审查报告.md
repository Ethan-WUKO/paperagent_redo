## 执行摘要

四项基础任务的冻结契约在其当前主运行路径中大体一致：Project REST 接口使用 `projectId` 与相对路径，不返回根目录；`ResolvedToolPolicy` 的空/deny-all 语义、模型工具暴露、普通 CHAT 兼容工具、上下文 `UNTRUSTED` envelope、chat/view=all 审计和 `ToolResult.success=false` 轨迹均有代码与定向测试支撑。V31 的 H2 Flyway 迁移、Project JPA 装配和 Spring Bean 装配通过。

但全聚合 `mvn test` 在 `yanban-paper` 失败：论文润色安全基线对同一“不安全占位符”场景期待 `FAILED_KEEP_ORIGINAL`，实现返回 `PROTECTION_REJECTED`。这既使聚合构建非绿，也说明既有论文润色状态契约未保持一致。因此本次结论为 **CHANGES_REQUIRED**。

## Findings

### P1

1. 论文润色安全基线状态与实现不一致，阻断全聚合构建。
   - 文件/行号：`yanban-paper/src/test/java/com/yanban/paper/quality/PaperPolishBaselineEvaluationTest.java:125,220-222`；`yanban-paper/src/main/java/com/yanban/paper/service/PaperSectionPolishService.java:133-146,205-207`。
   - 触发条件：baseline 的 `UNSAFE_DROP_PLACEHOLDER` 模式、`maxAttempts=1`。候选缺失占位符后进入保护拒绝分支，实际返回 `PROTECTION_REJECTED`，而基线冻结为 `FAILED_KEEP_ORIGINAL`。
   - 影响：`mvn test` 非绿，且“保留原文的失败状态”在论文润色基线中不再稳定。
   - 建议归属：论文润色/评测基线所有者。明确保留原文时两种状态的语义并让实现、评测 fixture 和既有取消路径一致；不要通过放宽断言掩盖差异。

### P2

1. 注解工具的实际执行器未接收已解析策略，策略约束依赖外层 provider/strategy 双重前置检查。
   - 文件/行号：`yanban-api/src/main/java/com/yanban/api/agent/AgentLangChain4jTools.java:389-397`；对比受治理执行入口 `yanban-core/src/main/java/com/yanban/core/tool/ToolRegistry.java:46-53`。
   - 触发条件：若未来新增调用路径直接调用 `AgentLangChain4jTools` 的注解方法（或跳过 `LangChain4jToolProvider`），其内部 `ToolRegistry.execute(call)` 不携带 `ResolvedToolPolicy.allowedTools`。
   - 影响：当前主运行路径在 `LangChain4jToolProvider.java:57-88` 与 `LangChain4jToolCallingStrategy.java:250-268` 已挡住绕过，定向测试也通过；但执行器本身尚未成为“同一 resolved policy”约束的独立边界。
   - 建议归属：工具治理与动态暴露所有者；在 Coordinator 接入前将注解桥接执行改为显式接收/验证 resolved allowlist，或限制其可见性并加入直接调用的回归测试。

### P0

无。

## 真实仍为 NOT_IMPLEMENTED 的能力

- Runtime Coordinator 及其唯一的 `ResolvedToolPolicy` 构造入口。
- Coordinator 级 success criteria/最终 outcome 验证；EvidenceLedger 不承担此职责。
- Runtime-managed async 的进度指纹、轮询最小间隔和无进展终止。
- 全量 model/tool/verification/finalization 的统一关联审计投影。
- Project 文件能力接入通用 Agent Runtime 的受治理 Project 工具（当前完成的是只读 REST/服务边界，不是 Coordinator 集成）。
- Linux 或具备创建链接权限的 Windows CI 上对真实符号链接/junction 的通过性验证；当前两个用例诚实 skipped，未计为失败或通过。

## 最终建议

**CHANGES_REQUIRED**

解除 P1、复跑全聚合构建并由主对话复审后，才能进入 Coordinator 阶段。P2 可与下一轮工具治理收口一并处理，但不得在 Coordinator 接入时依赖外层调用约定代替执行器约束。
