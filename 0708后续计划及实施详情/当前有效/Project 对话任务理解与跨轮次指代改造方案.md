# Project 对话任务理解与跨轮次指代改造方案

> 文档状态：问题总结与改造建议，尚未实施
> 创建日期：2026-07-23
> 适用范围：同一个 session 内的 Project 多轮对话、DIRECT / PLAN 路由、Planner、Plan 步骤执行与恢复、对话摘要、前端结果展示
> 明确不做：不同 session 之间恢复或继承任务；新会话不会自动加载旧会话的活跃任务。
> 保持不变的硬边界：Project 顶层仍只有 `DIRECT / PLAN_EXECUTE`；Candidate 未显式接受前始终 `NOT_APPLIED`；沙箱执行必须经过确认；模型不能通过理解结果为自己授予工具权限。

## 1. 文档目的

本文总结近期真实 Project 对话中暴露的问题，并给出一条从“理解用户请求”到“生成计划、执行、恢复和展示结果”的完整改造方向。

本轮讨论得出的核心结论是：

> 当前系统的主要缺口不是缺少 LLM 路由，而是没有把“当前用户请求 + 相关历史对话”稳定解析成一份可校验、可持续传递的任务定义。

如果任务目标、目标对象或用户约束在入口处没有被正确确定，后续 Planner、步骤工具权限、Candidate、沙箱执行、失败恢复和最终回答都会建立在错误基础上。现有安全边界能够阻止部分越权操作，但无法阻止上游生成语义错误、内部矛盾或偏离用户目标的计划。

本文只定义问题、目标方案、实施顺序和验收标准，不表示相关代码已经完成。

本文所称“跨轮次”仅指同一个会话内相邻或较早的 user/assistant 轮次，例如上一轮明确指定文件，下一轮使用“这份代码”“继续”“刚才那个结果”等指代。本文不设计跨会话任务账本，也不允许新会话自动继承旧会话任务。

## 2. 已观察到的真实问题

### 2.1 “过程详情”错误显示最终校验失败

等待用户确认时，真实状态可能是：

- `stopReason=WAITING_FOR_USER`
- `outcome=PARTIAL`
- Plan 已创建，但尚未开始沙箱执行

这不是执行失败，也不是最终校验失败。

当前过程摘要在缺少完整步骤投影、同时没有识别到特定 outcome 时，会落入兜底文案：

> 计划执行结果未通过最终校验。

因此该文案不是永远固定，但等待确认、部分投影或结构化步骤事实缺失时很容易稳定触发。它属于状态投影和展示语义错误，不是 LLM 生成错误。

### 2.2 执行计划出现在最终回答下面

当前前端把消息和 Plan 作为两个独立时间线对象，再按创建时间混合排序。

用户发送问题时，前端会立即创建临时 assistant 占位；Plan 由后端稍后创建。因此实时页面可能出现：

```text
用户问题
assistant 回答
执行计划
```

刷新后又可能因为服务器时间不同而重新排序。

此外，WebSocket 没有把 `WAITING_FOR_USER` 视为允许的受控中间状态，会将等待确认记录为失败，前端随后通过 HTTP 对账并较晚加载 Plan，进一步放大顺序问题。

期望的稳定顺序应为：

```text
用户问题
执行计划
唯一最终回答
```

Plan 应绑定到产生它的用户轮次，而不是只依赖时间戳猜测位置。

### 2.3 路由只选择策略，没有形成完整任务理解

当前 LLM Router 主要输出：

- `DIRECT` 或 `PLAN_EXECUTE`
- 任务结构
- 是否需要 Project 工具
- 是否需要多个步骤

这些信息足以决定走哪条运行路径，但不足以描述完整任务。它没有稳定输出并向后传递：

- 用户真正要完成的目标集合
- 每个目标作用于什么对象
- “这份代码”“刚才那个结果”等指代指向哪里
- 目标来源于哪一轮对话
- 用户要求的最终交付物
- 必须遵守的语言、范围、测试和格式约束
- 哪些信息仍有歧义

因此“路由选择了 PLAN”并不等于“系统已经理解了任务”。

### 2.4 路由和 Planner 没有获得同一份有效上下文

系统整体会构建上下文，但不同阶段收到的内容不一致：

- 路由器主要看到当前用户消息和有限的系统摘要/记忆提示，不直接消费完整普通对话。
- Planner 主要收到当前用户消息和受治理的长期记忆，不稳定接收最近普通对话中的目标对象。
- 路由器即使理解了某个指代，也没有结构化字段把该理解传给 Planner。
- Plan 步骤 Worker 主要看到 Plan 目标、当前步骤、依赖结果和受控工具，不负责重新恢复完整对话语义。

结果是上一轮已经明确的目标，在下一轮“这份代码”中没有形成可靠绑定。

### 2.5 缺少跨轮次“当前操作对象”

当前系统没有一份明确、可持续更新的活跃对象状态，例如：

- 当前讨论的是哪一个 Project 对象
- 它来自哪一轮用户明确指定
- 对应哪个 ProjectVersion
- 精确相对位置和内容哈希是什么
- 当前是否仍有效
- 用户最近要求对它做读取、修改还是执行

因此模型可能重新扫描整个 Project，并从多个候选对象中猜测一个目标。搜索结果越多，偏离上一轮真实目标的风险越高。

### 2.6 常见澄清问句被误判为修改命令

类似下面的请求：

> 你知道需要修改哪份代码吗？它的相对位置是什么？

本质是询问目标信息，不是要求立即修改。

但现有规则容易因为同时出现“修改”和“代码”而判定为代码变更意图，进而要求 Planner 必须包含 Candidate 步骤。Planner 若生成只读定位计划，就会被判为非法计划。

这不是少见边界情况。“你知道要改哪里吗”“刚才说的是哪一段”“先告诉我目标位置”都是正常连续对话。

### 2.7 失败恢复可能改变目标、语言和交付物

真实对话中已经出现：

- 原任务要求修改既有 Java 代码。
- 第一次读取步骤选错目标。
- 恢复计划漂移为创建新的 Python 脚本。
- 测试和返回代码也围绕新目标展开。

说明失败恢复没有被原始任务定义牢牢约束。恢复 Planner 能够重新解释任务，而不是只修复失败步骤。

### 2.8 步骤目标与步骤权限可能自相矛盾

已经出现步骤描述要求：

- 编写代码
- 创建新文件
- 执行测试
- 返回修改结果

但该步骤实际允许的工具只有读取、搜索和代码结构分析。

模型随后调用 Candidate 工具时被 `tool_permission_denied` 拒绝。拒绝本身是正确的安全行为，真正的问题是一个只有只读权限的步骤不应被接受为“写入和测试步骤”。

### 2.9 工具选择和预算会放大目标解析错误

“读取当前代码”步骤可能同时拥有结构分析、搜索和读取工具，但工具调用预算只有一次。模型如果先执行宽泛的多文件结构分析，就没有机会再精确读取真正目标。

因此需要区分：

- 目标尚未确定时的定位工具
- 目标已确定后的精确读取工具
- 修改步骤的 Candidate 工具
- 执行步骤的沙箱工具

不能只根据“代码任务”给出一个宽泛工具集合，再让模型在有限预算内自由猜测。

### 2.10 receipt 不能验证所有分析结论

沙箱 receipt 可以权威证明：

- 本次执行使用了哪个 Provider
- 状态、退出码和是否超时
- 捕获到的 stdout/stderr

但不能自动证明：

- 算法对所有输入都正确
- 时间复杂度分析正确
- 当前实现没有进一步优化空间

已经出现真实运行结果正确，但 LLM 对复杂度和“无需性能优化”的表述过于绝对。说明执行事实边界已经存在，但一般代码分析仍缺少针对性的语义校验。

## 3. 根因总结

| 问题 | 根因 | 性质 |
|---|---|---|
| 等待确认却显示最终校验失败 | 过程摘要没有正确识别 `WAITING_FOR_USER`，并在步骤事实缺失时使用悲观兜底 | 确定性实现缺陷 |
| Plan 位于回答下面 | 消息与 Plan 只按时间戳混排，没有绑定用户轮次 | 确定性前端建模缺陷 |
| WebSocket 等待确认被记录为失败 | 不同 API/事件层对受控等待状态定义不一致 | 状态协议缺陷 |
| “这份代码”指向错误目标 | 最近对话和上一轮精确目标没有形成可传递绑定 | 上下文与任务理解缺陷 |
| 澄清问句变成修改计划 | 关键词命中替代了句子动作和语气理解 | 意图识别缺陷 |
| Java 任务漂移成 Python 新文件 | 恢复阶段可以重新解释任务，核心目标未冻结 | 恢复契约缺陷 |
| 只读步骤尝试写入 | Planner 步骤语义与服务器工具权限没有做一致性校验 | 计划验证缺陷 |
| 运行成功但优化结论不准确 | receipt 只验证执行事实，一般推理缺少专项验证 | 回答质量缺口 |

总体判断：

> 当前核心安全边界基本有效，但任务理解、跨轮次目标绑定、Planner、恢复、步骤权限和前端状态展示之间存在集成断点。

## 4. 目标流程

建议将 Project 对话流程调整为：

```text
当前用户消息
  + 最近若干轮原始对话
  + 更早对话的滚动摘要
  + 当前活跃任务/对象状态
  + 当前 ProjectVersion 与有效 Evidence 锚点
                    ↓
              任务理解阶段
                    ↓
          指代解析、目标验证、歧义检查
                    ↓
             结构化任务定义
                    ↓
       服务器校验 requested effects 与权限
                    ↓
              DIRECT / PLAN_EXECUTE
                    ↓
          Planner 只负责拆解已确定任务
                    ↓
        执行、恢复、Final Synthesis 和展示
```

关键变化是：

> Router 和 Planner 之前必须先有一份可检查的任务理解；Planner 不再负责猜测用户到底想做什么。

## 5. 上下文组装方案

上下文组装是本轮改造的第一项生产能力，必须先于 TaskFrame、Router 和 Planner 改造。现有 ContextPackage、短期窗口、会话摘要和长期记忆虽然已经存在，但当前缺口是没有把同一会话内的最近 canonical 原文、较早摘要、活跃任务对象和 Project/Evidence 状态组装为一份可检查、可调试、可被后续阶段共同消费的输入。

### 5.1 必须传入的五类上下文

每次任务理解至少需要：

1. **当前用户消息**：不可改写的原始请求。
2. **最近原始对话**：保留最近若干轮 user/assistant canonical answer，用于解析“这份”“上一个”“继续”等指代。
3. **滚动对话摘要**：压缩更早历史，避免上下文无限增长。
4. **精确活跃状态**：当前目标、版本、路径、Evidence、Plan、Candidate、用户确认和未完成要求。
5. **当前 Project 状态**：当前 ProjectVersion、manifest 和目标是否仍有效。

### 5.2 最近原始对话与摘要不能互相替代

最近原始对话负责保留语言细节和指代关系；滚动摘要负责压缩长期语义。

摘要可能写成：

> 用户正在讨论一个全排列程序。

但不能替代以下精确信息：

- 目标相对位置
- ProjectVersion
- 文件哈希
- Candidate/Plan 标识
- 用户是否确认执行
- 哪些要求已完成或仍未完成

这些精确信息必须单独保存为结构化状态。

### 5.3 摘要更新规则

对话摘要应遵守：

- 只在一个用户轮次形成唯一 canonical answer 后更新。
- 等待确认、执行中和恢复中的中间文本不能写成已完成事实。
- Candidate 已生成不能摘要成 Project 已修改。
- 沙箱成功不能摘要成算法已证明普遍正确。
- Planner 计划创建某对象不能摘要成该对象已存在。
- 执行事实必须来自服务器状态和 receipt。
- 摘要始终是辅助语义数据，不是工具权限或 Evidence authority。

### 5.4 上下文预算

建议采用分层预算，而不是无限追加：

- 当前消息：完整保留。
- 最近对话：按最近完整轮次保留。
- 活跃任务和精确锚点：优先保留，不得因摘要截断丢失。
- 历史摘要：固定上限。
- 大型工具输出：只保留摘要和 Evidence 引用，需要时重新按引用读取。

## 6. 通用结构化任务定义

前述“代码修改任务字段”只是具体实例，不能作为唯一任务格式。建议使用通用外壳，允许不同领域扩展。

```text
TaskFrame
├─ rawRequest                 当前用户原文
├─ objectives[]               一个或多个目标
│  ├─ action                  读取、解释、比较、修改、执行、询问等
│  ├─ objectRef               作用对象
│  ├─ sourceTurn              来源轮次
│  ├─ successCriteria[]       可观察完成条件
│  └─ dependencies[]          目标依赖
├─ resolvedEntities[]         已解析对象
│  ├─ type                    代码、论文、实验结果、配置等
│  ├─ locator                 相对位置或受控标识
│  ├─ projectVersion
│  ├─ hash
│  └─ evidenceRefs[]
├─ constraints[]              用户明确限制
├─ deliverables[]             最终交付物及格式
├─ requestedEffects           用户请求的读取/修改/执行/外部访问
├─ preferences[]              语言和展示偏好
├─ ambiguities[]              尚未解决的指代、冲突和缺失信息
└─ contextSources[]           本次理解使用的上下文来源
```

注意：

- `requestedEffects` 表示模型理解到的用户请求，不等于服务器已经授权。
- 工具权限、Candidate、沙箱确认和写入权限仍由服务器独立计算。
- TaskFrame 是“语义提案 + 经验证目标”，不是 LLM 自我授权凭证。

### 6.1 不同任务共用同一外壳

**普通知识问答**

- objective：解释概念
- resolvedEntities：可以为空
- requestedEffects：无 Project 读取、无修改、无执行
- route：DIRECT

**只读代码理解**

- objective：解释当前实现
- resolvedEntities：当前版本中的精确代码对象
- requestedEffects：读取
- route：PLAN_EXECUTE

**跨材料比较**

- objectives：读取、比较、形成差异结论、输出表格
- resolvedEntities：论文、代码和实验结果
- requestedEffects：只读
- route：PLAN_EXECUTE

**代码修改与测试**

- objectives：修改、验证、返回结果
- constraints：保持语言、不得自动应用
- requestedEffects：Candidate + 沙箱执行
- route：PLAN_EXECUTE

## 7. 指代解析与歧义处理

### 7.1 解析顺序

对“这份代码”“刚才那个类”“上一个实验结果”应按以下顺序处理：

1. 查找当前活跃对象。
2. 查找最近用户明确指定的对象。
3. 查找最近 canonical answer 绑定的有效 Evidence。
4. 在当前 ProjectVersion 中验证对象仍存在且哈希匹配。
5. 如果只有一个有效候选，完成绑定。
6. 如果有多个候选、没有候选或版本已变化，停止并询问用户。

### 7.2 搜索的职责

搜索用于验证和消除歧义，不能让模型自由发明目标。

错误方式：

```text
用户说“这份代码”
→ 扫描整个 Project
→ 模型任选一个看起来相关的对象
```

正确方式：

```text
用户说“这份代码”
→ 从最近对话得到候选目标
→ 在当前 Project 中验证候选
→ 唯一且有效才继续
```

### 7.3 必须询问用户的情况

- 最近讨论了多个可能目标。
- 指代没有可追溯来源。
- 目标在当前版本中不存在。
- 历史目标和当前请求互相冲突。
- 用户要求“继续”，但存在多个未完成任务。
- 修改语言、目标或交付物需要改变原任务含义。

不能只依赖 LLM 自报置信度。必须结合候选数量、版本有效性和 Evidence 当前性做确定性判断。

## 8. 路由和 Planner 改造原则

### 8.1 路由前先完成最小任务理解

Router 不应只看当前句子中的关键词。至少先确认：

- 用户是在下命令、询问信息还是确认/拒绝。
- 是否依赖 Project 事实。
- 是否存在未解析目标。
- 是否请求修改或执行。
- 是否只需回答普通知识。

Router 可以继续由 LLM 提出建议，但服务器必须基于 TaskFrame 校验。

### 8.2 Router 结果必须携带任务定义

可以采用一次语义调用同时返回：

- TaskFrame
- 推荐 `DIRECT / PLAN_EXECUTE`
- 未解决歧义

服务器校验后再决定是否执行。不能让 Router 只返回四个策略字段，然后丢弃它对任务目标的理解。

### 8.3 Planner 只拆分任务，不重新解释任务

Planner 输入应包含：

- 已验证 TaskFrame
- 当前 ProjectVersion
- 精确目标锚点
- 用户约束和交付物
- 服务器计算后的工具权限上限

Planner 不得：

- 更换目标对象
- 更换语言或材料类型
- 新增用户未要求的写入
- 把澄清问题升级为修改
- 把修改既有对象漂移为创建无关新对象

### 8.4 计划一致性校验

计划持久化前必须验证：

- 每个步骤都能追溯到 TaskFrame 中的 objective。
- 写入步骤包含 Candidate 能力。
- 执行/测试步骤包含沙箱能力，并在执行前等待确认。
- 只读步骤不能以“写入成功”为完成条件。
- 目标相对位置与已解析目标一致。
- 用户要求保持 Java 时，计划不能无授权改为 Python。
- 测试步骤必须依赖 Candidate 或明确的当前版本。
- 返回结果步骤必须依赖真实读取或真实执行证据。

不满足时应重新规划或询问用户，而不是让步骤运行后再由权限层拒绝。

## 9. 失败恢复与 Reflection

### 9.1 冻结任务核心

Plan 创建时应冻结：

- 用户原始请求
- objectives
- resolvedEntities
- constraints
- deliverables
- requestedEffects
- 当前 ProjectVersion

恢复 Planner 只能修改：

- 工具参数
- 读取范围
- 步骤拆分
- 依赖顺序
- 在原目标范围内的替代方法

不能修改：

- 目标对象
- 用户要求的语言
- 修改与只读边界
- 最终交付物
- 是否需要测试
- 是否允许写入或执行

### 9.2 恢复前重新校验能力

恢复生成的每个步骤必须重新通过“步骤语义—工具权限—成功条件”一致性校验。

如果原失败原因是权限不足、等待确认、取消、超时或外部 Provider 不可用，不应通过 Reflection 重新解释任务来绕过。

### 9.3 目标无法继续时的行为

目标文件缺失、版本变化或歧义增加时，应暂停并向用户说明：

- 原目标是什么
- 当前发现了什么变化
- 需要用户选择什么

不得自动选择另一个对象继续。

---

## 10. 前端与状态协议修正

### 10.1 统一等待确认语义

API、WebSocket、Plan、UI 和 canonical answer 必须统一识别：

- `WAITING_FOR_USER` 是受控等待，不是执行失败。
- 等待确认时不得显示“未通过最终校验”。
- WebSocket 不应把等待确认发送为普通 runtime error。
- HTTP 对账只能用于恢复，不应成为每次等待确认的正常路径。

### 10.2 过程摘要由结构化状态生成

过程文案应优先依据：

- stopReason
- Plan lifecycle
- executionOutcome
- taskOutcome
- 是否等待确认
- 是否存在已完成/失败/跳过步骤

不得用一个悲观兜底覆盖明确的等待、取消、超时、失败或成功状态。

### 10.3 按用户轮次组织页面

前端应建立稳定轮次关系：

```text
Turn
├─ user message
├─ process summary
├─ plan card
└─ canonical assistant answer
```

排序依据应优先使用服务器关系：

- userMessageId
- planId
- assistantMessageId
- turn/run identity

时间戳只作为同类对象的次级排序，不能决定 Plan 属于哪个回答。

### 10.4 刷新和重启一致

实时页面、刷新后页面和 API 重启恢复后必须保持：

- Plan 位于对应回答上方。
- 唯一 user message。
- 唯一 assistant canonical answer。
- 等待确认、成功、失败、取消、超时状态一致。
- 不因 WebSocket 到 HTTP 对账产生第二轮执行。

## 11. 建议实施顺序

### 阶段 A：统一上下文包与临时调试展示

1. 复用并扩展现有 ContextPackage，明确组装当前用户原文、同一 session 最近若干个完整 canonical 轮次、较早滚动摘要、当前 ProjectVersion、有效 Evidence 引用和受治理长期偏好。
2. 上下文按完整轮次截取，不能留下孤立 user、assistant、tool 或 process 消息。
3. 最近原文和较早摘要必须同时存在；摘要不能覆盖路径、版本、hash、Plan、Candidate、确认状态等结构化事实。
4. 暂时在 Project 前端提供可展开的调试视图，展示实际送入任务理解阶段的上下文分区、来源、截断和丢弃原因；不得展示密钥、敏感环境变量、内部思维链或未授权数据。
5. 本阶段不实现 TaskFrame、不改变 Router/Planner 策略、不增加工具或权限。

这是当前最高优先级。没有统一且可观察的上下文输入，后续 TaskFrame 和指代解析无法可靠调试。

### 阶段 B：TaskFrame、活跃对象与同会话指代解析

1. 基于阶段 A 的上下文定义最小通用 TaskFrame。
2. 建立同一 session 内 active entity/target 绑定，保存来源轮次、ProjectVersion、路径、hash 和有效性。
3. 增加“这份代码”“刚才那个结果”“继续”等指代解析。
4. 区分命令、询问、确认、拒绝和澄清。
5. 目标不能唯一解析、版本失效或约束冲突时询问用户，不得猜测。
6. 不实现跨 session 任务恢复。

### 阶段 C：让 Router、Planner、步骤和恢复共用任务定义

1. Router 只基于已验证 TaskFrame 提出 `DIRECT / PLAN_EXECUTE`。
2. Planner 只根据 TaskFrame 拆步骤，不重新解释目标。
3. 校验步骤语义、工具权限、预算和成功条件一致。
4. 恢复与 Reflection 冻结目标、语言、约束和交付物。
5. 无法继续原目标时停止并询问，不得自动换目标。

### 阶段 D：短期工作记忆与摘要收口

1. 在同一 session 内持久化滚动摘要、活跃任务、活跃对象和未完成要求。
2. 摘要仅在唯一 canonical answer 形成后更新。
3. 等待确认、Candidate、执行事实和完成状态必须来自结构化状态，不能由摘要推断。
4. API 重启后恢复同一 session 的上下文状态；不跨 session 恢复任务。
5. 长期记忆只保存用户确认的稳定偏好或事实，不承担当前任务状态。

固定失败场景不作为阶段 A 的前置阻断项。每个阶段在实现对应能力时补充最小自动回归和真实多轮旅程，并逐步沉淀为固定验收集。

## 12. 固定验收场景

### 12.1 上下文与指代

1. 第一轮明确指定一个对象，第二轮说“这份代码”，必须绑定到同一对象。
2. 第一轮讨论两个对象，第二轮说“这个”，必须询问，不得猜测。
3. 第一轮目标在新 ProjectVersion 中消失，第二轮继续修改时必须提示目标已失效。
4. 滚动摘要后，精确目标位置、版本和 Evidence 仍然保留。
5. API 重启后，活跃任务和目标绑定能够恢复，不能依赖仅内存上下文。

### 12.2 意图理解

1. “修改这份代码”识别为修改请求。
2. “你知道需要修改哪份代码吗”识别为询问，不创建 Candidate。
3. “先告诉我目标位置，不要修改”必须保持只读。
4. “不要编译或运行”不能触发沙箱。
5. “不要修改，但请运行”只触发执行，不触发 Candidate。

### 12.3 Planner 与权限

1. 修改步骤必须拥有 Candidate 能力。
2. 测试步骤必须拥有沙箱能力并等待确认。
3. 只读步骤不能以写入成功为目标。
4. Java 目标失败恢复后仍然是同一 Java 目标。
5. 恢复步骤不得创建无关的新对象。
6. 工具权限不足时在计划持久化前失败，不应运行到中途才出现 permission denied。

### 12.4 前端与状态

1. 等待确认显示“等待用户确认”，不得显示最终校验失败。
2. Plan 稳定显示在对应 canonical answer 上方。
3. 实时、刷新、API 重启后三者顺序一致。
4. WebSocket 等待确认不触发普通失败。
5. 同一轮始终只有一个 user、一个 Plan 状态源和一个 assistant canonical answer。

### 12.5 证据与最终回答

1. 当前 ProjectVersion 的路径、哈希、范围支持代码事实。
2. 版本变化后旧 Evidence 失效。
3. receipt 只证明本次执行状态和捕获输出。
4. 复杂度、普遍正确性等结论不得冒充 receipt 事实。
5. 失败、超时、取消不能被 Final Synthesis 改写为成功。

---

## 13. 明确不采用的方案

1. 不把无限增长的完整对话全部塞给每一个 LLM。
2. 不只靠自由文本摘要保存精确目标。
3. 不让 Planner 自己决定工具权限。
4. 不让 LLM 的高置信度替代目标唯一性和版本校验。
5. 不在目标不明确时扫描整个 Project 后任选一个对象。
6. 不增加新的 Project 顶层策略；仍保持 `DIRECT / PLAN_EXECUTE`。
7. 不因为任务理解改造而放宽 Candidate、沙箱、Evidence 和 Final Synthesis 的既有安全边界。

---

## 14. 完成定义

只有同时满足以下条件，才能认为本轮问题真正解决：

1. 普通连续对话中的跨轮次指代可以稳定解析。
2. 无法唯一解析时系统会询问，而不是猜测。
3. Router、Planner、步骤执行和恢复共享同一份已验证任务定义。
4. Planner 和恢复不能改变用户目标、语言、约束和交付物。
5. 步骤语义、工具权限和成功条件一致。
6. 对话摘要控制上下文长度，同时不丢失精确目标和权威状态。
7. 等待确认在所有 API/事件/UI 层保持一致。
8. Plan 卡片稳定位于对应回答上方。
9. 真实执行事实、代码解释和模型推断继续严格分层。
10. 上述固定验收场景形成自动测试和真实浏览器旅程。

最终目标不是要求 LLM 永远理解正确，而是：

> 让 LLM 基于足够且相关的上下文提出任务理解；让服务器验证目标是否真实、唯一、当前有效；让不确定性在执行前显式暴露；让后续所有阶段都服从同一份已确认任务定义。
