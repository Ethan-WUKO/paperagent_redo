# 科研工具与结构化索引契约（第一版）

状态：`FROZEN_CONTRACT / READ_ONLY_IMPLEMENTATION_ACCEPTED`。本文与 `yanban-core` 中的 `com.yanban.core.research` 继续作为不可变数据契约、JSON Schema 和确定性校验的权威来源。契约冻结后，五个工具已经通过受治理的 parser、ToolExecutor、ToolRegistry 和 Runtime 接入完成只读实现；实现记录已归档到 `历史归档/实施记录/第一批低风险只读科研工具实施记录.md`。后续扩展不得反向放宽 trusted scope、相对路径、Evidence、预算、错误或去重边界。

## 信任与边界

- Runtime 单独注入并保管 trusted `projectId`、`userId`、capability 和 Project root；它们不属于任何模型工具参数 schema，也不进入普通 Evidence 或索引投影。
- 模型只能提交下面表中的参数。所有文件名均为规范化 Project 相对路径；绝对路径、盘符路径、反斜杠和 `.`/`..` 路径段会被拒绝。
- 论文、代码、BibTeX、CSV、配置和报告的文本均是 `UNTRUSTED_PROJECT_CONTENT`，只能作为数据呈现，永远不能成为系统指令、权限或策略来源。
- `ProjectVersionRef`、`FileManifestEntry`、hash、范围、parser 版本和 `SERVER_ATTESTED_METADATA` 是可审计来源元数据。`ProjectVersionRef` 兼容现有 Project manifest 的裸 64 位 SHA-256（构造时归一化为小写），也可使用安全 namespaced token；拒绝 Windows/UNC 路径及 `file:`/HTTP URI。`ParserVersionRef` 只允许如 `latex-parser@1` 的可移植 identity/version token；拒绝 drive/UNC/反斜杠、正斜杠路径、`file:`/HTTP URI、控制字符和首尾空白。两者都以 JSON string 投影，绝不含 Project id 或主机路径。
- 每次调用都以 `toolName + trusted ProjectVersionRef + canonical JSON 参数 SHA-256` 形成稳定 deduplication key；JSON 对象键排序，数组顺序保持不变。

## 五个只读工具

所有工具的契约 descriptor 都是 `sideEffect=READ_ONLY`、`confirmation=NEVER`、`repeat=DENY_SAME_INPUT`、`async=SYNC`、`requiredPermission=research:project-read`。契约定义本身不授予可见性；生产 Runtime 只能在 `PROJECT + READ_ONLY + resolved allowedTools` 求交通过后动态暴露相应实现。

| 工具 | 模型可控输入及上限 | 预算上限 | 结构化输出 items |
| --- | --- | --- | --- |
| `project_latex_outline` | 必填 `relativePaths`（最多 20）；可选 `includeFormulaReferences:boolean` | 200 items / 300 Evidence / 2 MB | section、label、formula-reference、citation、float 目录 |
| `project_bibtex_audit` | 必填 `relativePaths`（最多 20）；可选 `includeUnusedEntries:boolean` | 300 / 400 / 2 MB | 条目、重复、缺字段、未引用与缺失 citation key |
| `project_code_symbols` | 必填 `relativePaths`（最多 30）；可选 `symbolQuery`（1–160 字符）、`includeDependencies:boolean` | 500 / 600 / 3 MB | class/function/entry point/parameter/dependency |
| `project_experiment_summary` | 必填 `relativePaths`（最多 30）；可选 `metricNames`（最多 30）、`maxRowsPerFile`（1–500） | 300 / 400 / 5 MB | config、CSV metrics、report、result-file summary |
| `project_cross_material_search` | 必填 `query`（1–200 字符）；可选 `relativePaths`（最多 50）、`maxMatches`（1–100） | 100 / 300 / 5 MB | 论文、代码、配置和实验材料间的概念链接 |

共同输出 schema 的必填字段是 `status`、`items`、`evidenceRefs`、`partial`、`truncated`、`parseFailed`。`status` 为 `COMPLETE`、`EMPTY`、`PARTIAL`、`TRUNCATED` 或 `PARSE_FAILED`；`ResearchToolOutcome` 直接序列化为这些字段，预算和 retryability 是服务端/审计字段而不出现在模型结果中。错误码使用稳定的 `ResearchToolErrorCode`，其中仅 `TRANSIENT_PROJECT_IO` 可重试。每个 Evidence 具有 `projectVersion`、`relativePath`、`fileHash`、`range.startLine/endLine`、`parserVersion`、`trustLabel`。

`items` 不是通用自由文本对象。五个工具分别使用封闭的 `LatexOutlineItem(kind, identifier?, detail, content)`、`BibtexAuditItem(issue, citationKey?, detail, content)`、`CodeSymbolItem(kind, qualifiedName, dependencyReference?, content)`、`ExperimentSummaryItem(assetType, metricName?, value, content)` 与 `CrossMaterialLinkItem(concept, relation, linkedEvidence, content)`；`?` 为仅可省略、不可使用空字符串占位的 NON_NULL 字段。每个 `content` 都固定为 `text + evidence + UNTRUSTED_PROJECT_CONTENT`，所有对象 `additionalProperties=false`。因此输出能表达 section/formula/citation、BibTeX issue、symbol/dependency、metric/config 及跨材料链接，而不允许任意 Map 或未声明字段。

状态一致性同样是契约：`COMPLETE` 必须有 item 且无 error；`EMPTY` 不得有 item、Evidence 或 error；`PARTIAL` 必须有 item/Evidence 并使用 `PARTIAL_RESULT`；`TRUNCATED` 必须有 item/Evidence 并使用 `RESULT_TRUNCATED`；`PARSE_FAILED` 必须是无 item/Evidence 的 `PARSER_FAILURE`。每个 item 的 `content.evidence` 都必须被 outcome 的 `evidenceRefs` 覆盖；跨材料 item 的全部 `linkedEvidence` 也必须进入 envelope，供后续 Completion Verifier 使用。

审计投影只包含工具名、dedup key、状态、错误与 retryable、Evidence 和预算用量，不包含用户/项目内部标识、capability、绝对路径或源文本。

## 最小索引模型

`FileManifestEntry` 作为 `ProjectVersionRef + ProjectRelativePath + FileHash + sizeBytes` 的稳定引用。每个索引实体通过 `IndexedProvenance` 保留其 manifest、源行范围、parserVersion、`CURRENT`/`STALE`/`INVALIDATED`、partial、truncated 和信任标签。

| 索引 | 契约内容 |
| --- | --- |
| `LatexSectionIndex` | section id、标题、层级、来源 |
| `LatexFormulaReferenceLink` | formula label 到 reference label 的双端来源 |
| `BibEntryIndex` | citation key、字段名、citation usage 来源 |
| `CodeSymbolIndex` | symbol id、种类、限定名、参数名、来源 |
| `CodeDependencyReference` | symbol/module 依赖边与来源 |
| `ExperimentAssetIndex` | 配置、指标、报告或结果资产摘要与来源 |
| `CrossMaterialLinkIndex` | 跨论文/代码/实验至少两份材料的概念关系及新鲜度 |

示例：`CrossMaterialLinkIndex("learning rate", [paper/main.tex:41-45, configs/train.yaml:3-3, reports/run-17.csv:2-20], "configured-by", CURRENT)`。三个来源各自携带同一 Project baseline 的版本、相对路径、hash 和 parserVersion；实际文本仍以不可信数据处理。

## 后续 Worker 接入点与禁止事项

后续实现 Worker 可在受控 Runtime 解析 trusted scope 后：调用 `ResearchToolContract.callKey`（它会先校验输入再生成去重键）、使用 `ResearchBudget.validate` 约束结果，并将 parser 产物转换为以上索引和 `ResearchEvidenceRef`。路径必须已经规范化：空路径数组、数组非文本/空白/越长项、小数整数参数、带首尾空白路径、Windows ADS 冒号和 C0 控制字符都会被拒绝；中文及正常内部空格路径仍可用。`ResearchRuntimeScope` 是 fail-closed server-only 类型，普通 ObjectMapper 序列化会失败。接入时才可实现各工具 executor，并需单独复审 ToolRegistry/ResolvedToolPolicy/Completion Verifier 兼容性。
