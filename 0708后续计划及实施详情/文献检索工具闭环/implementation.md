# 文献检索工具闭环实施方案

## 一、目标

当前系统已经有一个文献检索工具，后续不应该一开始就拆成很多新工具，而是先把这个工具升级成一个可复用的底层能力。

最终目标是：

1. 支持普通文献检索。
2. 支持论文中的补引用。
3. 支持相关工作和综述补充。
4. 支持替换不合适的文献。
5. 支持用户确认、拒绝、收藏、加入文献库。
6. 支持结果沉淀，后续论文润色、普通对话、项目分析都可以复用。
7. 支持评测回归，避免每次修改后质量倒退。

一句话来说：

文献检索工具不能只做到“搜到几篇论文”，而要做到“知道为什么搜、搜出来怎么用、用户是否认可、后续能不能复用”。

## 二、总体设计

后续仍然以一个文献检索工具作为底层能力，但在它上面增加四层能力。

### 1. 意图识别层

先判断用户到底想做什么。

常见任务类型包括：

1. 普通找文献。
2. 找最新文献。
3. 找经典文献。
4. 补一句话的引用。
5. 补相关工作或综述。
6. 替换不合适的引用。
7. 检查已有参考文献是否完整。
8. 判断已有文献是否过旧、跑题或重复。

这一步的作用是防止所有需求都被当成“关键词搜索”。

### 2. 检索策略层

底层还是调用同一个文献检索能力，但不同任务要使用不同策略。

普通找文献：

1. 更重视整体相关性。
2. 可以返回代表性论文、综述论文、最新论文。
3. 输出可以比较宽。

补引用：

1. 更重视某篇论文能不能支撑某一句话。
2. 检索问题应该来自目标句子的真实含义。
3. 推荐数量要少而准。

补综述：

1. 更重视覆盖不同研究类别。
2. 结果需要分组，而不是简单排序。
3. 既要有经典论文，也要有近期论文。

替换文献：

1. 更重视新候选文献是否比旧文献更合适。
2. 需要比较旧引用和新候选。
3. 不应该只因为年份旧就替换经典文献。

### 3. 判断和解释层

检索结果不能直接给用户，需要二次判断。

每篇候选文献至少要判断：

1. 是否真的相关。
2. 适合解决什么问题。
3. 是否能支撑目标句子。
4. 是否适合补某个综述类别。
5. 是否比原引用更合适。
6. 是否已经存在于用户文献库或当前论文参考文献中。
7. 元数据是否完整。
8. 是否存在不确定性。

输出时必须让用户看得懂“为什么推荐这篇”。

### 4. 闭环沉淀层

用户对推荐结果的操作要被记录下来。

至少支持以下状态：

1. 未处理。
2. 已收藏。
3. 已加入文献库。
4. 已拒绝。
5. 不再推荐。
6. 已用于某篇论文。
7. 用户已读。

这些状态要影响后续推荐。

例如：

1. 用户拒绝过的论文，不应该反复出现。
2. 用户标记“已经有了”的论文，不应该当成新文献推荐。
3. 用户收藏过某方向文献，后续类似主题可以适当提高权重。
4. 用户确认用于某篇论文的文献，后续润色时应该能识别出来。

## 三、统一输出格式

不管是普通找文献、补引用、补综述，还是替换文献，最终都应该使用统一结果结构。

每条文献结果建议包含：

1. 标题。
2. 作者。
3. 年份。
4. 来源。
5. 摘要。
6. 链接。
7. DOI 或其他稳定标识。
8. BibTeX。
9. 推荐理由。
10. 匹配对象。
11. 任务类型。
12. 可信程度。
13. 是否已存在于用户文献库。
14. 是否已存在于当前论文参考文献。
15. 用户操作状态。
16. 排序分数。
17. 风险提示。

其中“匹配对象”非常重要。

不同任务下含义不同：

1. 普通找文献：匹配用户的检索主题。
2. 补引用：匹配论文中的某一句话。
3. 补综述：匹配某个研究类别。
4. 替换文献：匹配需要替换的旧引用和所在句子。

## 四、补引用实施方案

### 目标

补引用不是随便推荐几篇论文，而是给论文中缺少证据支撑的位置找到合适文献。

### 流程

1. 读取论文内容。
2. 找出可能缺引用的句子。
3. 判断这些句子是否真的需要引用。
4. 把每个缺引用句子转换成检索问题。
5. 调用文献检索工具。
6. 对候选论文进行二次判断。
7. 输出推荐文献和支撑理由。
8. 用户确认是否采用。
9. 用户确认后生成引用建议和 BibTeX。
10. 记录这篇文献已用于该论文。

### 缺引用句子的判断标准

优先识别以下类型句子：

1. 陈述已有研究结论的句子。
2. 介绍某个方法、模型、理论来源的句子。
3. 描述某个领域趋势的句子。
4. 提到某个数据集、指标、评测方法的句子。
5. 对比不同方法优劣的句子。
6. 提到“近年来”“已有研究表明”“广泛应用”等表达的句子。

暂时不需要补引用的句子：

1. 作者自己的方法描述。
2. 本文实验结果。
3. 常识性过渡句。
4. 小节引导句。
5. 已经有合适引用支撑的句子。

### 输出要求

每个缺引用位置输出：

1. 原句。
2. 为什么认为这里需要引用。
3. 建议检索主题。
4. 推荐文献列表。
5. 每篇文献能支撑原句的哪一部分。
6. 是否建议新增引用。
7. 是否需要用户人工确认。

### 边界条件

1. 不自动把引用插入正文，必须用户确认。
2. 一句话最多推荐 1 到 3 篇文献。
3. 找不到合适文献时要明确说找不到，不要硬凑。
4. 元数据不完整的文献不能直接作为正式引用。
5. 摘要和主题不匹配的文献不能推荐。
6. 只支持普通论文内容，不处理完整项目编译和复杂 LaTeX 工程。

## 五、补综述实施方案

### 目标

补综述不是给一堆论文列表，而是判断当前相关工作缺少哪些研究类别，然后按类别推荐文献。

### 流程

1. 读取论文题目、摘要、引言、相关工作。
2. 提取论文主题。
3. 分析当前相关工作已经覆盖哪些类别。
4. 判断还缺少哪些类别。
5. 对每个缺失类别生成检索问题。
6. 调用文献检索工具。
7. 对结果进行分组和排序。
8. 输出每个类别下的推荐文献。
9. 给出放置建议。
10. 用户确认后，生成引用清单和可选的段落修改建议。

### 常见综述类别

不同学科会有差异，但普通论文可以先按以下类别判断：

1. 基础理论。
2. 代表性方法。
3. 最新进展。
4. 相关应用场景。
5. 数据集和评测方法。
6. 局限性和挑战。
7. 对比方法。
8. 综述性论文。

### 输出要求

补综述结果应该按类别展示：

1. 缺失类别名称。
2. 为什么需要补这一类。
3. 建议放在相关工作的哪个位置。
4. 推荐文献。
5. 每篇文献的作用。
6. 是否是经典文献。
7. 是否是近期文献。

### 边界条件

1. 不直接生成大段综述替换原文，先给结构建议。
2. 不追求推荐数量多，优先保证类别完整。
3. 经典文献不能因为年份旧就被排除。
4. 新文献不能因为年份新就自动排前面。
5. 应用论文不能随便当成领域综述核心文献。
6. 用户没有确认前，不修改正文和参考文献。

## 六、替换不合适文献实施方案

### 目标

替换文献不是简单找更新的论文，而是判断旧引用是否真的不适合，以及是否有更合适的替代文献。

### 流程

1. 找到论文中某个引用所在的句子。
2. 获取被引用文献的信息。
3. 判断旧文献和当前句子的匹配程度。
4. 判断旧文献是否存在问题。
5. 如果确实有问题，再生成替换检索问题。
6. 调用文献检索工具。
7. 比较新候选文献和旧引用。
8. 输出替换建议。
9. 用户确认后，生成替换后的引用建议和 BibTeX。
10. 记录旧文献和新文献的处理状态。

### 判断旧文献是否不合适

常见问题包括：

1. 主题不匹配。
2. 支撑的是宽泛背景，不支撑当前具体观点。
3. 领域不一致。
4. 方法不一致。
5. 年份过旧且不是经典文献。
6. 元数据不完整。
7. 引用重复。
8. 引用质量明显不足。

### 输出要求

替换建议必须包含：

1. 原句。
2. 当前引用。
3. 当前引用的问题。
4. 是否建议替换。
5. 替换候选文献。
6. 新文献比旧文献更合适的原因。
7. 是否建议保留旧文献。

### 边界条件

1. 不要只因为旧就替换。
2. 经典文献可以保留。
3. 如果旧引用合适，要明确说不建议替换。
4. 替换必须经过用户确认。
5. 新文献必须比旧文献更贴近当前句子。
6. 不确定时只能给候选，不要自动替换。

## 七、针对现有缺陷的修改思路

### 缺陷一：任务边界不清楚

问题：

普通找文献、补引用、补综述、替换文献容易混在一起。

解决思路：

1. 增加任务类型识别。
2. 每次调用检索前先确认任务类型。
3. 不同任务使用不同检索策略和排序权重。
4. 输出结果中保留任务类型。
5. 前端按任务类型展示不同信息。

验收标准：

用户说“帮我补这句话的引用”时，系统不能只返回论文列表，必须说明每篇文献支撑哪句话。

### 缺陷二：推荐结果没有沉淀

问题：

用户选了什么、拒绝了什么、已经用过什么，系统后续不知道。

解决思路：

1. 给推荐结果增加用户状态。
2. 增加文献库沉淀能力。
3. 用户收藏、拒绝、加入文献库后写入记录。
4. 后续排序读取这些记录。
5. 同一论文在不同任务中出现时合并状态。

验收标准：

用户拒绝过的论文，后续不能继续作为强推荐反复出现。

### 缺陷三：排序容易跑偏

问题：

只靠关键词容易推荐看似相关、实际不贴题的论文。

解决思路：

1. 把排序拆成多项分数。
2. 至少包含主题匹配、方法匹配、领域匹配、年份、来源质量、元数据完整度、是否重复、用户反馈等维度。
3. 补引用时提高“能否支撑目标句子”的权重。
4. 补综述时提高“是否覆盖缺失类别”的权重。
5. 替换文献时提高“是否比旧引用更贴合”的权重。

验收标准：

关键词相同但领域明显不一致的论文，不能排在强推荐位置。

### 缺陷四：外部论文源不稳定

问题：

外部接口可能慢、失败、返回空结果或元数据不完整。

解决思路：

1. 支持主来源和备用来源。
2. 支持本地文献库兜底。
3. 支持缓存结果。
4. 返回结果时标记来源状态。
5. 检索失败时明确提示失败原因。
6. 支持部分结果返回。

验收标准：

某个外部来源失败时，系统不能直接报错结束，应该返回可用的部分结果和清楚的失败说明。

### 缺陷五：元数据不完整

问题：

缺作者、年份、来源或稳定链接的文献不适合直接引用。

解决思路：

1. 给每篇文献增加可信程度。
2. 根据信息完整度分级。
3. 元数据不足的文献只能作为阅读候选。
4. 生成 BibTeX 前做字段检查。
5. 不完整时提示用户人工确认。

验收标准：

元数据不足的文献不能被系统静默当成正式引用。

### 缺陷六：前端展示不足

问题：

如果只展示标题列表，用户很难判断哪篇值得用。

解决思路：

1. 把结果页改成决策面板。
2. 展示标题、作者、年份、来源、推荐理由、适合用途、可信程度。
3. 补引用场景展示支撑的原句。
4. 补综述场景按类别分组。
5. 替换文献场景展示新旧对比。
6. 提供收藏、拒绝、加入文献库、复制引用等操作。

验收标准：

用户不用点开论文，也能初步判断这篇文献为什么被推荐。

### 缺陷七：论文润色和文献推荐没有打通

问题：

论文润色不能真正知道哪些地方需要文献，也不能把推荐结果对应到具体句子。

解决思路：

1. 在论文润色中增加引用缺口分析。
2. 对每个缺口生成检索问题。
3. 调用统一文献检索能力。
4. 推荐结果绑定到具体句子或段落。
5. 用户确认后生成引用建议。
6. 引用建议和 BibTeX 进入论文润色结果包。

验收标准：

论文润色输出中，用户能看到“哪句话缺引用、推荐哪篇、为什么推荐”。

### 缺陷八：没有用户反馈闭环

问题：

系统无法通过用户反馈变得更准确。

解决思路：

1. 增加最小反馈按钮：有用、不相关、已经有了、不再推荐。
2. 每次反馈写入记录。
3. 后续检索时读取反馈记录。
4. 排序时降低被拒绝文献或类似文献权重。
5. 提高用户收藏方向的相关文献权重。

验收标准：

用户的明确反馈能影响下一次推荐结果。

### 缺陷九：没有真实评测闭环

问题：

固定样例只能证明基础逻辑，不能证明真实论文源和真实任务稳定可用。

解决思路：

1. 保留固定评测集作为回归基线。
2. 增加真实外部检索测试。
3. 增加人工抽查用例。
4. 增加跑偏案例。
5. 增加中文需求找英文论文案例。
6. 每次修改检索、排序、提示词后都跑评测。

验收标准：

文献检索相关修改不能只看接口通过，必须看推荐质量没有明显下降。

### 缺陷十：长任务影响对话体验

问题：

深度文献检索可能耗时很久，不适合阻塞普通对话。

解决思路：

1. 区分快速推荐和深度检索。
2. 快速推荐直接返回少量结果。
3. 深度检索走任务模式。
4. 任务模式支持进度、取消、部分结果、最终报告。
5. 对话里可以先返回“已找到一部分”，后续再补完整结果。

验收标准：

用户发起深度检索后，可以看到进度，可以取消，也可以查看已获得的部分结果。

## 八、开发顺序

### 第一阶段：任务类型识别

目标：

让系统先知道用户到底要做哪类文献任务。

需要做：

1. 定义任务类型。
2. 增加任务识别逻辑。
3. 给每种任务定义输入字段。
4. 给每种任务定义输出要求。
5. 在工具调用前写入任务类型。

完成标准：

同一个文献检索工具可以根据不同任务返回不同形式的结果。

### 第二阶段：统一输出结构

目标：

所有文献任务返回统一结构，方便前端展示、后端存储和后续复用。

需要做：

1. 定义统一结果对象。
2. 补充推荐理由字段。
3. 补充匹配对象字段。
4. 补充可信程度字段。
5. 补充用户状态字段。
6. 补充 BibTeX 和元数据检查字段。

完成标准：

普通找文献、补引用、补综述、替换文献都能用同一套结果结构表达。

### 第三阶段：补引用闭环

目标：

先把最核心、最容易落地的补引用做好。

需要做：

1. 识别缺引用句子。
2. 生成检索问题。
3. 调用文献检索。
4. 判断候选文献是否支撑该句。
5. 输出引用建议。
6. 用户确认后生成 BibTeX。
7. 记录该文献已用于论文。

完成标准：

系统能清楚告诉用户：哪句话缺引用，推荐哪几篇，每篇支撑什么。

### 第四阶段：补综述闭环

目标：

让系统能判断相关工作缺哪些类别，并按类别推荐文献。

需要做：

1. 分析论文主题。
2. 分析已有相关工作覆盖情况。
3. 识别缺失类别。
4. 按类别检索文献。
5. 按类别输出推荐。
6. 给出放置建议。

完成标准：

系统输出的不再是一堆论文，而是“缺什么类别、为什么缺、每类补哪几篇”。

### 第五阶段：替换文献闭环

目标：

支持判断已有引用是否不合适，并推荐更合适替代文献。

需要做：

1. 获取引用所在句子。
2. 获取旧文献信息。
3. 判断旧文献是否合适。
4. 生成替换检索问题。
5. 检索候选文献。
6. 比较新旧文献。
7. 输出替换建议。
8. 用户确认后记录替换结果。

完成标准：

系统不仅能推荐新文献，还能说明为什么新文献比旧文献更合适。

### 第六阶段：用户反馈和文献库沉淀

目标：

让系统从一次性检索工具变成长期可复用的文献助手。

需要做：

1. 增加用户操作状态。
2. 增加文献库记录。
3. 增加用户反馈记录。
4. 后续检索读取这些记录。
5. 排序时考虑用户历史反馈。

完成标准：

用户认可、拒绝、用过的文献都会影响后续推荐。

### 第七阶段：评测回归

目标：

保证每次修改都不会让文献推荐质量倒退。

需要做：

1. 建立固定测试集。
2. 建立真实外部检索测试。
3. 建立补引用测试。
4. 建立补综述测试。
5. 建立替换文献测试。
6. 每次修改后记录评测结果。

完成标准：

文献检索相关改动必须有质量评测结果，而不是只看接口是否成功。

## 九、最小可行版本

如果要尽快做出第一版，建议只做以下内容：

1. 增加任务类型：普通找文献、补引用、补综述、替换文献。
2. 统一文献结果结构。
3. 先实现补引用流程。
4. 前端展示推荐理由和用户操作按钮。
5. 记录用户收藏、拒绝、已使用。
6. 增加基础评测用例。

第一版不做：

1. 不做完整 Project 系统。
2. 不做复杂 LaTeX 工程编译。
3. 不做自动改正文。
4. 不做全自动替换引用。
5. 不接入过多外部工具。
6. 不追求覆盖所有学科。

## 十、最终验收标准

当以下条件满足时，可以认为文献检索工具闭环基本完成：

1. 用户普通找文献时，系统能返回高相关论文、推荐理由和 BibTeX。
2. 用户补引用时，系统能指出具体句子、推荐文献、解释支撑关系。
3. 用户补综述时，系统能指出缺失类别，并按类别推荐文献。
4. 用户替换文献时，系统能比较旧引用和新候选。
5. 用户拒绝过的文献不会反复强推荐。
6. 用户确认使用的文献会进入文献库或论文结果包。
7. 外部检索失败时系统能返回部分结果和失败说明。
8. 元数据不完整的文献不会被静默当成正式引用。
9. 前端能清楚展示推荐原因、可信程度和用户操作。
10. 每次修改后都有评测结果记录。

## 十一、给 AI 开发者的 issue 拆分规则

本方案可以交给 AI 开发者拆分 issue，但必须按以下规则拆，避免多个对话同时修改同一块核心逻辑，或者把边界做歪。

### 1. 一个 issue 只解决一个明确问题

每个 issue 必须包含：

1. 要解决的问题。
2. 不做什么。
3. 涉及的后端能力。
4. 涉及的前端能力。
5. 数据结构是否变化。
6. 是否需要数据库迁移。
7. 是否需要新增评测。
8. 完成标准。
9. 验证方式。

不能写成“完善文献检索闭环”这种大 issue。

### 2. 必须先做基础结构，再做业务场景

推荐顺序：

1. 任务类型识别。
2. 统一输出结构。
3. 用户反馈状态。
4. 补引用闭环。
5. 补综述闭环。
6. 替换文献闭环。
7. 前端决策面板。
8. 评测回归。

其中第 1、2、3 项是基础能力，后面的能力都依赖它们。

补引用、补综述、替换文献不要在统一输出结构完成前大规模开发，否则后续会反复返工。

### 3. 并行开发边界

可以并行：

1. 后端统一输出结构和前端展示原型可以并行，但前端只能先按约定字段做，不要自己发明字段。
2. 评测用例整理可以和业务开发并行。
3. 用户反馈接口可以和补引用逻辑并行，但必须使用统一的文献标识。
4. 文献来源稳定性和缓存兜底可以单独并行。

不建议并行：

1. 两个对话同时改文献排序核心逻辑。
2. 两个对话同时改统一输出结构。
3. 两个对话同时改数据库迁移。
4. 两个对话同时改同一个前端结果页。
5. 在补引用未稳定前，同时做自动正文修改。

### 4. 每个 issue 的完成标准必须可验证

完成标准不能只写“功能完成”。

应该写成：

1. 输入什么。
2. 系统输出什么。
3. 哪些字段必须存在。
4. 哪些行为不能发生。
5. 哪些测试必须通过。
6. progress.md 里必须记录什么。

例子：

用户输入“帮我给这句话补引用”，系统必须输出原句、推荐文献、推荐理由、支撑关系、可信程度、BibTeX 状态，并且不能自动修改正文。

### 5. 每个 issue 合并前必须更新 progress.md

每个开发对话完成任务后，必须在 progress.md 记录：

1. issue 编号。
2. 分支名称。
3. 完成内容。
4. 关键改动。
5. 测试结果。
6. 未解决问题。
7. 是否影响后续 issue。

如果没有更新 progress.md，视为该 issue 没有闭环。

## 十二、建议拆分的第一批 issue

### Issue 1：定义文献任务类型

目标：

让系统能区分普通找文献、补引用、补综述、替换文献、检查已有参考文献。

需要做：

1. 定义任务类型枚举或等价结构。
2. 定义每类任务需要的输入字段。
3. 在调用文献检索前写入任务类型。
4. 给普通对话和论文润色都预留调用入口。

不做：

1. 不做复杂前端页面。
2. 不做自动修改论文正文。
3. 不做完整补引用流程。

验收标准：

系统能根据用户请求识别任务类型，并把任务类型传递到文献检索流程中。

### Issue 2：统一文献结果结构

目标：

让所有文献任务都返回同一套结果结构。

需要做：

1. 定义统一结果对象。
2. 包含文献基础信息。
3. 包含推荐理由。
4. 包含匹配对象。
5. 包含可信程度。
6. 包含用户状态。
7. 包含 BibTeX 状态。
8. 包含风险提示。

不做：

1. 不改复杂排序逻辑。
2. 不做补综述业务。
3. 不做替换文献业务。

验收标准：

普通文献推荐结果可以使用统一结构返回，旧接口不能出现明显破坏。

### Issue 3：增加文献用户反馈状态

目标：

记录用户对推荐文献的操作。

需要做：

1. 支持收藏。
2. 支持拒绝。
3. 支持已引用。
4. 支持不再推荐。
5. 支持加入文献库。
6. 后续检索结果能显示已有状态。

不做：

1. 不做复杂个性化推荐。
2. 不做大规模用户画像。
3. 不做自动替换引用。

验收标准：

用户拒绝或收藏某篇文献后，下一次结果中能看到对应状态。

### Issue 4：实现补引用最小闭环

目标：

对单句或小段文本给出可确认的引用建议。

需要做：

1. 接收目标句子或段落。
2. 判断是否需要引用。
3. 生成检索问题。
4. 调用文献检索。
5. 判断候选文献是否支撑目标句子。
6. 输出推荐文献、支撑关系和 BibTeX 状态。
7. 用户确认后记录已用于论文。

不做：

1. 不自动插入正文。
2. 不批量重写论文。
3. 不处理复杂 LaTeX 工程。

验收标准：

给定一句需要引用的论文句子，系统能返回 1 到 3 篇候选文献，并说明每篇支撑该句子的哪一部分。

### Issue 5：实现补综述最小闭环

目标：

分析相关工作缺失类别，并按类别推荐文献。

需要做：

1. 读取题目、摘要、引言、相关工作。
2. 提取论文主题。
3. 判断已有相关工作覆盖类别。
4. 找出缺失类别。
5. 按类别调用文献检索。
6. 输出类别、缺失原因、推荐文献和放置建议。

不做：

1. 不直接生成大段综述替换原文。
2. 不自动修改正文。
3. 不追求覆盖所有学科。

验收标准：

系统能输出“缺哪几类文献、每类为什么需要补、每类推荐哪些文献”。

### Issue 6：实现替换文献最小闭环

目标：

判断旧引用是否不合适，并推荐更合适的替代文献。

需要做：

1. 获取引用所在句子。
2. 获取旧文献元数据。
3. 判断旧文献是否合适。
4. 生成替换检索问题。
5. 检索候选文献。
6. 比较新旧文献。
7. 输出是否建议替换和替换理由。

不做：

1. 不只因为年份旧就替换。
2. 不自动删除旧引用。
3. 不自动修改正文。

验收标准：

系统能说明旧引用的问题，以及新候选文献为什么更适合；如果旧引用合适，必须能输出“不建议替换”。

### Issue 7：前端文献结果决策面板

目标：

让用户能看懂推荐文献为什么有用，并完成确认、拒绝、收藏等操作。

需要做：

1. 展示标题、作者、年份、来源。
2. 展示推荐理由。
3. 展示匹配对象。
4. 展示可信程度。
5. 展示 BibTeX 状态。
6. 支持收藏、拒绝、加入文献库、复制引用。
7. 补引用场景展示原句。
8. 补综述场景按类别展示。
9. 替换文献场景展示新旧对比。

不做：

1. 不做复杂项目空间。
2. 不做自动改论文。
3. 不自行定义和后端不一致的字段。

验收标准：

用户不打开论文详情，也能初步判断推荐文献是否值得采用。

### Issue 8：文献检索评测回归

目标：

保证文献检索相关修改有质量检查。

需要做：

1. 普通找文献测试。
2. 补引用测试。
3. 补综述测试。
4. 替换文献测试。
5. 跑偏案例测试。
6. 中文需求找英文论文测试。
7. 外部来源失败兜底测试。

不做：

1. 不要求一次覆盖所有学科。
2. 不把评测只做成接口是否成功。

验收标准：

每次修改文献检索、排序、提示词或输出结构后，都能跑评测并记录结果。
