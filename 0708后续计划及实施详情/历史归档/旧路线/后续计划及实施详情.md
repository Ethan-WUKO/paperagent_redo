# 0708 后续计划及实施详情

## 总目标

把当前系统从“论文润色、文献检索、知识库、聊天工具各自能用”，升级成一个稳定的科研 Agent 工作台。

短期不做完整 Project，不做复杂代码仓库协同，不做完整论文项目编译沙箱。先把四件事做稳：

1. 文献推荐独立可用。
2. 普通论文润色质量闭环。
3. 通用 Agent 具备基本工具选择、执行、反思、可观测能力。
4. 评测体系成为每次改动的硬门槛。

核心原则：

1. 不再盲目堆功能。
2. 每一步都必须有边界、验收、评测和回归。
3. 先稳定底层能力，再扩通用 Agent，最后再做 Project。

## 阶段一：文献推荐能力收口

### 目标

让“帮我找论文”成为一个独立能力，而不是只服务论文润色流程。

### 需要做什么

1. 明确文献推荐输入。
   - 用户问题。
   - 研究领域。
   - 关键词。
   - 期望数量。
   - 年份范围。
   - 是否需要最新文献。
   - 是否已有参考文献列表。

2. 明确文献推荐输出。
   - 标题。
   - 作者。
   - 年份。
   - 来源。
   - 链接或 DOI。
   - 摘要。
   - 推荐理由。
   - 相关性评分。
   - 是否可能重复。
   - 是否适合引用。

3. 文献推荐结果必须可解释。
   - 不能只给列表。
   - 要说明为什么推荐。
   - 要说明它支持用户哪个研究问题或论点。

4. 做好去重。
   - 如果用户已有参考文献，要区分“已经存在的文献”和“真正新增的候选文献”。
   - 信息不足、需要人工确认的文献必须标注风险。

5. 文献推荐要能被三个地方复用。
   - 聊天里“帮我找论文”。
   - 论文润色里的引用建议。
   - 未来 Project 的项目文献库。

### 边界条件

1. 不自动把推荐文献插入论文正文。
2. 不生成不存在的论文。
3. 信息不完整的文献必须标注风险。
4. 不为了凑数量推荐明显不相关的论文。
5. 文献推荐任务可以是后台任务，不要求一次对话同步等到全部完成。

### 验收标准

1. 用户问“帮我找 5 篇某方向论文”，系统能返回真实、相关、可解释的结果。
2. 至少保留一套固定测试集，每次改排序、提示词、去重逻辑都要跑。
3. 推荐结果要有分数和理由。
4. 重复文献不能作为多篇独立推荐出现。

## 阶段二：普通论文润色质量闭环

### 目标

先把普通 LaTeX 论文润色做好。暂时不追求复杂多文件项目、模板编译、代码实验联动。

### 需要做什么

1. 修复用户配置不生效的问题。
   - 用户设置的润色标准、最大轮数、内部尝试次数，后端必须真正使用。
   - 前端显示什么，后端就按什么执行。

2. 统一评分尺度。
   - 如果界面是 0 到 100 分，后端审核也必须用 0 到 100 分。
   - 不能前端显示 75 分，后端实际按 0.7 判断。

3. 每个章节都要记录处理结果。
   - 是否成功润色。
   - 是否保留原文。
   - 失败原因。
   - 是否触发格式保护。
   - 是否模型返回异常。
   - 是否审核未通过。

4. 自动生成润色前后差异。
   - 用户要能看到系统到底改了什么。
   - 不能只给最终稿。

5. 明确“修改意见”和“文献推荐”的区别。
   - 修改意见：针对正文表达、结构、论证、段落逻辑。
   - 文献推荐：候选文献，不默认写入正文。
   - 如果要把修改意见合入正文，必须经过用户确认。

6. 建立“采纳修改意见后重新生成”的闭环。
   - 用户点采纳后，系统才把对应修改应用到最终稿或生成新版稿件。
   - 用户点拒绝后，不能继续应用该建议。

7. 保持 LaTeX 保护。
   - 公式、引用、label、ref、图表环境、章节命令，默认不能被模型乱改。

8. 编译验证暂时作为后续能力。
   - 当前只要求普通论文润色稳定。
   - 完整项目编译、沙箱、模板文件检查，后续放到 Project 或沙箱能力中做。

### 边界条件

1. 不支持复杂多文件 LaTeX 项目的完整编译。
2. 不自动改图。
3. 不自动新增实验数据。
4. 不自动强化论文 claim。
5. 不自动插入未确认文献。
6. 如果模型不确定，应保留原文并说明原因，而不是硬改。

### 验收标准

1. 普通 `.tex` 论文能完成润色。
2. 每章都有状态和失败原因。
3. 用户配置真实生效。
4. 用户能看到改动差异。
5. 用户采纳或拒绝建议会影响后续产物。
6. 系统不会把英文论文整段翻译成中文，除非用户明确要求翻译。

## 阶段三：论文润色评测集

### 目标

不要靠感觉判断论文润色变好了还是变坏了。

### 需要做什么

1. 固定一批样例论文。
   - 英文论文。
   - 中文论文。
   - 含公式论文。
   - 含引用论文。
   - 结构比较普通的论文。

2. 为每篇样例定义检查项。
   - 是否保留原意。
   - 是否保护引用和公式。
   - 是否真的有语言改善。
   - 是否没有新增虚假内容。
   - 是否没有破坏 LaTeX 结构。

3. 每次改润色提示词、审核逻辑、修复逻辑，都必须跑评测。

4. 输出评测报告。
   - 章节成功率。
   - 保留原文比例。
   - 审核失败原因分布。
   - 平均耗时。
   - 模型调用次数。
   - 明显质量问题。

### 边界条件

1. 评测集不要求覆盖所有复杂模板。
2. 不用一开始追求自动判断所有学术质量。
3. 但必须覆盖“不能改坏”的底线。

### 验收标准

1. 有可重复运行的评测命令。
2. 有固定报告。
3. 改动前后能比较质量变化。
4. AI 开发者提交相关改动时必须附评测结果。

## 阶段四：通用 Agent 运行时收口

### 目标

让普通对话不再只是“模型加一堆工具”，而是有策略、有边界、有过程记录的 Agent。

### 需要做什么

1. 建立统一的运行入口。
   - 每次用户发消息，系统先判断任务类型。
   - 可能类型包括：直接回答、单步工具调用、多步计划任务、长任务、需要用户确认的任务。

2. 建立策略选择。
   - 简单问题直接回答。
   - 需要查资料才调用搜索或文献工具。
   - 多步骤任务才进入计划模式。
   - 涉及修改论文、写入知识库、删除内容、调用高风险工具时先确认。

3. 工具选择不能完全交给模型。
   - 系统要先筛工具，再给模型。
   - 模型不应该每次都拿到全部工具。

4. 普通对话只能产生一个最终回答。
   - 工具调用过程、检索过程、反思过程可以作为过程事件记录。
   - 这些过程事件不能变成多个助手气泡污染聊天。

5. 增加反思检查。
   - 有没有回答用户问题。
   - 证据是否足够。
   - 是否用了不该用的工具。
   - 工具失败时有没有说明限制。
   - 多步任务是否有遗漏。

6. 多步任务要有计划、执行、检查、失败恢复。
   - 计划不是摆设。
   - 要能跟踪每一步是否完成。
   - 失败后可以修复、降级或请求用户补充信息。

### 边界条件

1. 不一次性重写现有聊天系统。
2. 不删除现有论文、文献、知识库接口。
3. 不强制所有问题都走计划模式。
4. 不让模型自由调用所有工具。
5. 不把内部思考、工具中间输出直接展示成最终回答。

### 验收标准

1. 稳定常识问题不调用搜索。
2. 用户明确要求搜索时才搜索。
3. 用户要求找论文时使用文献推荐。
4. 多步骤任务可以生成计划并执行。
5. 工具失败时有清晰降级说明。
6. 一轮用户输入只产生一个最终回答。

## 阶段五：工具系统升级

### 目标

把工具从“能调用”升级成“可治理的能力”。

### 每个工具需要补充的信息

1. 什么时候应该用。
2. 什么时候不应该用。
3. 是否只读。
4. 是否会修改用户资产。
5. 是否需要用户确认。
6. 成本高不高。
7. 预计耗时。
8. 超时时间。
9. 失败是否重试。
10. 每轮最多调用几次。
11. 输出格式是什么。
12. 结果是否用户可见。
13. 风险等级。

### 优先整理的工具

1. 网页搜索。
2. 知识库搜索。
3. 文献推荐。
4. 文献检索任务。
5. 论文润色任务查询。
6. 论文润色结果读取。
7. 论文任务取消。
8. 文件系统工具。
9. GitHub 工具。

### 边界条件

1. 暂时不接入太多外部 MCP。
2. 高风险工具默认不开放给普通对话。
3. 文件读写必须有白名单。
4. 修改用户资产必须确认。
5. 长任务工具只返回任务编号和状态，不同步阻塞到完成。

### 验收标准

1. 每个核心工具都有清晰说明和调用预算。
2. 系统能根据任务筛选工具。
3. 工具结果格式统一，方便模型理解。
4. 工具失败不会导致整轮对话崩溃。

## 阶段六：MCP 和外部工具扩展

### 目标

在工具治理稳定之后，再扩更多外部工具。

### 优先顺序

1. 文件读取和项目分析。
2. GitHub 仓库、issue、PR 查询。
3. 学术数据库相关工具。
4. 浏览器或网页读取工具。
5. 文档处理工具。
6. 后续沙箱执行工具。

### 边界条件

1. MCP 不默认全部暴露。
2. 每个 MCP 工具必须经过白名单和权限控制。
3. 文件系统工具必须限制根目录。
4. GitHub 写操作必须用户确认。
5. 沙箱执行能力后续再做，不放在当前论文润色阶段。

### 验收标准

1. MCP 注册失败不能影响普通聊天。
2. 用户能知道某个工具不可用的原因。
3. 工具调用过程可追踪。
4. 高风险操作有确认流程。

## 阶段七：模型配置体验确认

### 目标

确认用户自定义模型能力真的可用，而不是只有代码入口。

### 需要做什么

1. 检查登录用户是否可以添加自定义模型。
2. 检查能否填写模型地址、模型名、API Key。
3. 检查能否测试连接。
4. 检查能否选择为默认模型。
5. 检查普通聊天是否使用该模型。
6. 检查计划任务是否使用该模型。
7. 检查文献推荐和论文润色是否使用正确模型。
8. 检查失败提示是否清楚。

### 边界条件

1. 游客不能修改模型配置。
2. API Key 不能明文返回前端。
3. 删除模型不能影响历史会话展示。
4. 自定义模型不保证兼容所有工具调用格式，不兼容时要提示。

### 验收标准

1. 普通用户能自助添加、测试、使用模型。
2. 失败时能知道是地址错、Key 错、模型名错，还是接口不兼容。
3. 所有主要链路使用的模型来源清楚可见。

## 阶段八：长期记忆和上下文治理

### 目标

让系统逐渐了解用户，但不能乱记、乱用、污染回答。

### 需要做什么

1. 区分聊天历史、短期上下文、长期记忆。
2. 长对话不能把全部历史塞给模型。
3. 需要有会话摘要。
4. 长期记忆只保存稳定、有价值的信息。
5. 用户可以查看、删除、修正长期记忆。
6. 每次注入长期记忆都要能解释来源。
7. 记忆不能压过当前用户指令。

### 边界条件

1. 不把所有聊天内容自动写入长期记忆。
2. 不保存敏感信息。
3. 用户删除的记忆不能再注入。
4. 低置信度记忆不能当事实用。
5. Project 级记忆以后再做，先做用户级记忆。

### 验收标准

1. 长对话不会无限膨胀上下文。
2. 用户能看到系统记住了什么。
3. 删除或修正记忆后，后续回答不再使用旧记忆。
4. 记忆注入有调试记录。

## 阶段九：统一任务和事件体验

### 目标

用户能清楚知道任务跑到哪、是否失败、是否等待自己、是否可以取消。

### 需要做什么

1. 统一长任务状态。
   - 等待中。
   - 运行中。
   - 等待用户。
   - 取消请求中。
   - 取消中。
   - 完成。
   - 失败。
   - 已取消。

2. 统一任务事件。
   - 创建时间。
   - 当前阶段。
   - 进度。
   - 最近事件。
   - 错误原因。
   - 是否需要用户操作。
   - 是否有可下载产物。

3. 取消必须是安全取消。
   - 用户点击取消后，系统先进入“正在取消”。
   - 到安全点后再真正结束。

4. 部分结果要标注清楚。
   - 已取消任务中间生成的内容不能当最终结果。

### 边界条件

1. 不强杀线程。
2. 不在取消后继续写最终产物。
3. 不把失败任务显示成完成。
4. 不因为刷新页面重复提交任务。
5. 不要求一次性迁移所有旧接口。

### 验收标准

1. 用户能看到任务状态。
2. 用户能取消任务。
3. 取消后不会生成正式最终稿。
4. 重连或刷新不会重复最终回答。
5. 任务失败原因清楚。

## 阶段十：论文润色 UI 重做

### 目标

让用户一眼看懂当前任务，而不是在一堆卡片里找信息。

### 建议布局

1. 左侧：任务列表。
   - 查看历史任务。
   - 查看运行中任务。
   - 查看失败任务。
   - 查看已取消任务。

2. 中间：当前任务流程。
   - 上传。
   - 解析。
   - 结构确认。
   - 文献推荐。
   - 润色。
   - 导出。

3. 右侧：当前阶段结果。
   - 结构确认。
   - 文献推荐。
   - 章节 diff。
   - 失败原因。
   - 下载入口。

4. 顶部：任务总状态。
   - 耗时。
   - 模型。
   - 阶段。
   - 是否需要用户操作。

### 边界条件

1. 不在 UI 重做时顺便大改后端逻辑。
2. 不引入 Project 概念干扰当前流程。
3. 不展示用户不需要理解的内部调试信息。
4. 调试信息放高级入口。

### 验收标准

1. 用户知道现在跑到哪一步。
2. 用户知道自己是否需要操作。
3. 用户知道结果是否完整。
4. 用户能找到下载结果和失败原因。
5. 多任务切换不混乱。

## 阶段十一：图表理解，暂不自动改图

### 目标

先让系统读懂图表并给建议，不做自动图片编辑。

### 需要做什么

1. 解析论文里的图、表、标题、编号和正文引用。
2. 抽取图表说明。
3. 检查正文是否准确描述图表。
4. 生成图表改进建议。
5. 标注哪些建议需要用户人工处理。

### 边界条件

1. 不自动改图片。
2. 不自动重画实验图。
3. 不根据图表编造实验结论。
4. 不把图表理解作为当前论文润色的阻塞项。

### 验收标准

1. 能列出论文中的主要图表。
2. 能指出 caption 太弱、正文未解释、引用不清等问题。
3. 建议以人工审核为主。

## 执行纪律

后续每个开发任务都必须写清楚：

1. 本次做什么。
2. 本次不做什么。
3. 用户可见行为有什么变化。
4. 影响哪些链路。
5. 需要跑哪些测试。
6. 是否需要 Agent、RAG、论文润色评测。
7. 风险是什么。
