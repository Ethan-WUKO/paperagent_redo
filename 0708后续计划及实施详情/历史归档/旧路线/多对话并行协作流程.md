# 多对话并行协作流程

## 目标

后续开发允许同时开启多个 AI 对话，但必须用 GitHub issue、独立分支、独立 PR 来管理，避免多个对话互相覆盖代码、重复实现、扩大范围或把不稳定改动直接合进主分支。

核心目标：

1. 每个对话只处理一个明确 issue。
2. 每个 issue 有清晰边界、验收标准和非目标。
3. 每个对话在独立分支开发。
4. 每个分支通过 PR 合并。
5. 主分支始终保持可运行、可测试、可回滚。

---

## 总体流程

标准流程如下：

```text
总计划
  -> 拆 GitHub issues
  -> 每个 issue 开一个 AI 对话
  -> 每个对话创建独立分支
  -> 开发和本地验证
  -> 提交 PR
  -> 人工/AI review
  -> 按顺序合并到 main
  -> 其他分支同步 main
```

禁止流程：

```text
多个对话同时改 main
多个对话处理同一个 issue
多个对话同时大改同一批核心文件
没有测试结果就合并
没有非目标说明就任由 AI 扩大范围
```

## Issue 拆分规则

### 一个 issue 只做一件事

每个 issue 必须足够小，能在一次对话或一个 PR 中完成。

推荐粒度：

1. 论文润色参数真正生效。
2. 章节失败原因记录。
3. 润色前后 diff 输出。
4. 修改意见采纳后重新生成产物。
5. 文献推荐输入输出协议收口。
6. 文献推荐评测报告补充。
7. 工具选择策略第一版。
8. 单轮只生成一个最终回答。
9. 工具治理信息补充。
10. 自定义模型体验验证。

不推荐粒度：

1. “重构论文润色系统”。
2. “完善 Agent 能力”。
3. “接入更多 MCP”。
4. “重做所有 UI”。
5. “实现 Project”。

这些太大，必须继续拆。

### 每个 issue 必须包含的内容

每个 GitHub issue 至少写清楚：

```text
背景：
- 为什么要做这个？

目标：
- 本 issue 完成后用户或系统会得到什么？

范围：
- 本 issue 具体做哪些点？

非目标：
- 本 issue 明确不做哪些点？

影响模块：
- 可能涉及哪些模块或页面？

验收标准：
- 什么情况算完成？

测试要求：
- 需要跑哪些测试？
- 是否需要 Agent/RAG/论文润色 eval？

风险：
- 可能破坏什么？

依赖关系：
- 是否依赖其他 issue 先完成？
```

### Issue 优先级

建议优先级：

1. P0：阻塞主流程或会导致错误结果的问题。
2. P1：重要能力闭环，影响用户体验或产品可信度。
3. P2：增强体验、调试能力、可观测能力。
4. P3：后续扩展或低优先级优化。

当前推荐先开 P0/P1：

1. 论文润色参数真正生效。
2. 章节失败原因记录。
3. 润色 diff。
4. 文献推荐输入输出协议收口。
5. 工具选择策略第一版。
6. 单轮最终回答治理。

## 分支规则

### 每个 issue 单独分支

分支命名建议：

```text
codex/issue-编号-简短描述
```

示例：

```text
codex/issue-101-paper-polish-params
codex/issue-102-section-failure-reasons
codex/issue-103-paper-polish-diff
codex/issue-104-literature-contract
codex/issue-105-tool-policy-v1
```

### 分支创建规则

1. 开始开发前，先从最新 `main` 创建分支。
2. 不要从另一个未合并 feature 分支继续开新分支，除非明确存在依赖。
3. 如果确实依赖另一个 PR，必须在 issue 和 PR 里写清楚。

### 分支同步规则

当某个 PR 合并进 `main` 后，其他正在开发的分支必须尽快同步最新 `main`。

同步后要重新跑相关测试。

## 多对话分工规则

### 每个对话只处理一个 issue

给 AI 对话的开场指令建议：

```text
你只处理 GitHub issue #xxx。
请先阅读 issue 内容和相关文档。
本次只实现 issue 范围内的内容，不要顺手重构其他模块。
如果发现需要扩大范围，先汇报，不要直接改。
完成后给出变更摘要、测试结果和风险说明。
```

### 不同对话避免改同一批核心文件

并行开发时要人为错开区域。

推荐并行组合：

1. 一个对话改论文润色后端参数。
2. 一个对话改文献推荐评测。
3. 一个对话改 Agent 工具策略。
4. 一个对话改文档和 issue 模板。

不推荐并行组合：

1. 两个对话同时改论文润色编排器。
2. 两个对话同时改同一个前端页面。
3. 一个对话重构工具系统，另一个对话同时接新工具。
4. 一个对话改模型配置，另一个对话改所有模型调用链路。

### 冲突高风险文件

这些文件或模块如果多个对话同时修改，冲突概率高，需要谨慎：

1. 论文润色编排核心。
2. Agent 普通对话服务。
3. 工具注册和工具策略。
4. 设置页。
5. 主路由和布局。
6. 数据库 migration。
7. 通用任务状态和事件模型。

处理方式：

1. 同一时间只允许一个 issue 修改同一核心文件。
2. 如果必须同时改，先指定主 PR，另一个 PR 等主 PR 合并后再 rebase 或重开。
3. 涉及 migration 时，必须确认版本号不冲突。

## 开发前检查清单

每个 AI 对话开始前必须做：

1. 确认当前 issue 编号和目标。
2. 确认当前分支不是 `main`。
3. 确认工作区是否已有未提交改动。
4. 阅读相关设计文档。
5. 搜索相关代码。
6. 判断是否会和其他正在跑的 issue 冲突。
7. 给出简短实施计划。

如果发现工作区已有无关改动：

1. 不要回滚。
2. 不要覆盖。
3. 如果无关，忽略。
4. 如果影响当前任务，先汇报。

## 开发中规则

### 严格控制范围

AI 开发者只能做 issue 范围内的改动。

禁止：

1. 顺手重构。
2. 顺手改 UI 风格。
3. 顺手改命名。
4. 顺手升级依赖。
5. 顺手扩大数据库结构。
6. 顺手把旧接口删掉。

允许：

1. 为完成 issue 做必要的小范围调整。
2. 补必要测试。
3. 补必要文档。
4. 在 PR 里记录发现但未处理的问题。

### 遇到范围扩大时

如果开发中发现必须做额外工作：

1. 先停止扩大实现。
2. 在对话里汇报：
   - 发现了什么问题。
   - 为什么原 issue 范围不够。
   - 有哪些处理方案。
   - 推荐方案是什么。
3. 用户确认后再继续。

## 测试和评测规则

### 最低测试要求

每个 PR 至少要有对应测试或验证说明。

常见类型：

1. 后端逻辑改动：跑相关 Maven 测试。
2. 前端改动：跑前端 build。
3. Agent 策略改动：跑工具选择或 Agent eval。
4. RAG 改动：跑 RAG eval。
5. 文献推荐改动：跑文献推荐 eval。
6. 论文润色改动：跑论文润色相关测试或样例评测。
7. 文档改动：至少检查 markdown 和 diff。

### 需要 eval 的情况

以下改动必须跑或补 eval：

1. Agent 策略选择。
2. 工具调用策略。
3. 工具 schema。
4. RAG 检索。
5. 文献排序和去重。
6. 论文润色 prompt。
7. 论文润色 review 或 repair。
8. 长期记忆抽取和注入。
9. 可能导致多气泡、重复回答、事件混乱的改动。

### PR 中必须写测试结果

PR 描述必须包含：

```text
测试：
- 已执行：...
- 结果：...

未执行：
- ...
- 原因：...

风险：
- ...
```

禁止只写：

```text
看起来正常
简单测了一下
应该没问题
```

## PR 规则

### PR 标题

建议格式：

```text
[Issue #xxx] 简短说明
```

示例：

```text
[Issue #101] Make paper polish thresholds effective
```

### PR 内容模板

每个 PR 必须写：

```text
## 背景

## 本次改动

## 非目标

## 用户可见变化

## 测试结果

## 风险和回滚

## 关联 issue
Closes #xxx
```

### PR 不能做什么

1. 不能混多个 issue。
2. 不能混无关格式化。
3. 不能混大范围重构。
4. 不能混无关 UI 改动。
5. 不能删除兼容接口，除非 issue 明确要求。

---

## Review 规则

Review 时先看风险，不先看代码风格。

检查顺序：

1. 是否超出 issue 范围。
2. 是否破坏现有用户流程。
3. 是否有测试。
4. 是否有 eval。
5. 是否有回滚方式。
6. 是否引入数据兼容问题。
7. 是否影响其他并行分支。
8. 是否需要更新文档。

### 高风险 PR 必须人工确认

这些 PR 不能只让 AI 自己合：

1. 数据库 migration。
2. 任务状态机。
3. 工具权限。
4. 文件读写。
5. 模型 API Key 处理。
6. 论文最终产物生成。
7. 删除或重命名公共接口。
8. 涉及用户数据隔离。

---

## 合并顺序规则

建议合并顺序：

1. 文档和协议类 PR。
2. 后端底层能力 PR。
3. 测试和评测 PR。
4. 后端产品行为 PR。
5. 前端展示 PR。
6. UI 重做 PR。

不要把 UI PR 合在底层能力之前，否则 UI 可能展示不存在或不稳定的数据。

### 推荐近期合并顺序

1. 文献推荐协议和评测基线。
2. 论文润色参数生效。
3. 章节失败原因。
4. 润色 diff。
5. 采纳建议后重新生成。
6. 工具选择策略。
7. 单轮最终回答治理。
8. 运行 trace。
9. UI 调整。

---

## 冲突处理规则

### 小冲突

如果只是 import、注释、相邻小代码冲突：

1. 开发者自行解决。
2. 重新跑相关测试。
3. PR 里说明同步过 main。

### 大冲突

如果冲突涉及核心逻辑：

1. 不要让 AI 盲目合并。
2. 暂停该分支。
3. 先看已合并 PR 的设计意图。
4. 决定是重做当前 PR，还是拆分当前 PR。

### 重复实现

如果两个 PR 做了类似能力：

1. 选一个作为主线。
2. 另一个 PR 关闭或改成补充。
3. 不要把两套实现都合进去。

---

## 数据库 Migration 规则

如果多个对话都需要新增 migration：

1. 必须提前认领版本号。
2. PR 合并前确认版本号没有冲突。
3. 后合并的 PR 如果版本冲突，必须重命名 migration。
4. migration 必须有 H2 测试资源同步，除非项目明确不需要。
5. migration 不能随意修改已经合并的历史 migration。

---

## 文档同步规则

涉及以下内容时必须同步文档：

1. 新增用户可见功能。
2. 改变任务状态。
3. 改变工具行为。
4. 改变模型配置。
5. 新增 eval。
6. 新增重要限制。
7. 改变论文润色流程。

文档可以放在：

1. 当前计划文件夹。
2. `docs/design`。
3. `docs/evaluation/reports`。
4. `memory-bank`。

原则：

1. 当前执行计划放在本文件夹。
2. 长期设计沉淀放 `docs/design`。
3. 评测结果放 `docs/evaluation/reports`。
4. 历史决策和阶段总结放 `memory-bank`。

## 每个对话结束时必须汇报

每个 AI 对话结束时，必须输出：

```text
完成内容：
- ...

修改文件：
- ...

测试：
- ...

未完成：
- ...

风险：
- ...

建议下一步：
- ...
```

如果创建了分支、提交、PR，也必须说明：

```text
分支：
提交：
PR：
```

## 并行开发建议分组

### 第一组：可并行

1. 文献推荐协议文档。
2. 文献推荐评测补充。
3. 论文润色参数生效。
4. 自定义模型体验验证。

这些改动区域相对独立。

### 第二组：谨慎并行

1. 章节失败原因。
2. 润色 diff。
3. 采纳建议后重新生成。

这些都可能碰论文润色核心，最好按顺序来。

### 第三组：不要和其他核心改动并行

1. Agent 工具选择策略。
2. 单轮最终回答治理。
3. 运行 trace。
4. 工具治理信息。

这些会影响普通聊天主链路，建议一次只合一个。

### 第四组：后置

1. MCP 扩展。
2. 论文润色 UI 重做。
3. 图表理解。
4. Project。
5. 沙箱。

这些等底层能力稳定后再做。

---

## 推荐第一批 Issue

建议先开这些 issue：

1. `论文润色参数真实生效`
   - 修复润色标准、最大轮数、内部尝试次数未真正生效的问题。

2. `论文润色章节失败原因记录`
   - 每章记录成功、失败、保留原文、保护规则触发、模型失败等原因。

3. `论文润色 diff 输出`
   - 生成并展示润色前后差异。

4. `文献推荐输入输出协议收口`
   - 明确聊天、论文润色、未来 Project 共用的推荐协议。

5. `文献推荐评测基线补充`
   - 固定评测报告和运行方式。

6. `Agent 工具选择策略第一版`
   - 不再默认把所有工具给模型。

7. `单轮对话只产生一个最终回答`
   - 工具过程进入 trace，不污染最终回答。

8. `自定义模型体验验证`
   - 确认用户添加、测试、选择、使用自定义模型的完整链路。

---

## 最重要的纪律

1. 先开 issue，再开对话。
2. 一个对话只做一个 issue。
3. 一个 issue 一个分支。
4. 一个分支一个 PR。
5. PR 必须有测试和风险说明。
6. 不同时大改同一核心文件。
7. 不把未验证能力合进主分支。
8. 每合一个 PR，其他分支同步主分支。
9. 发现超范围问题，先汇报再继续。
10. 主分支永远保持可运行。
