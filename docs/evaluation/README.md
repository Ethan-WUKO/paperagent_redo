# 研伴 Agent v1 评测说明

本文档定义 v1 上线前的评测方法。目标不是证明 Agent 很聪明，而是确认它在客户场景里可靠、可解释、不会卡死、不会越权。

## 评测资产

- `v1-eval-suite.md`：人工与半自动评测用例。
- `launch-checklist.md`：上线前门禁清单。
- `current-assessment-2026-06-30.md`：当前版本评估报告。

## 推荐执行顺序

1. 运行自动化测试。
2. 构建前端。
3. 启动本地依赖与后端。
4. 上传 `docs/kb-samples/*.md` 到知识库。
5. 按 `v1-eval-suite.md` 执行 P0/P1 用例。
6. 按 `launch-checklist.md` 判断是否允许对客户试用。
7. 将结果记录到评估报告。

## 自动化基线

在项目根目录运行：

```powershell
mvn test
```

在前端目录运行：

```powershell
pnpm build
```

这些测试只覆盖工程正确性，不覆盖真实模型质量。真实模型质量必须执行 `v1-eval-suite.md` 的人工评分项。

## 评分规则

每个人工评测项按 0-5 分记录：

| 分数 | 含义 |
|---|---|
| 5 | 完全满足预期，表达清晰，有必要来源或限制说明 |
| 4 | 基本满足，只有轻微遗漏或表达问题 |
| 3 | 部分满足，但需要人工补救 |
| 2 | 主要目标失败，只有少量可用信息 |
| 1 | 输出严重错误、空泛或不可执行 |
| 0 | 无输出、报错、卡死、越权、严重幻觉 |

## v1 上线门槛

- P0 用例必须全部通过。
- 自动化测试必须通过。
- 前端构建必须通过。
- RAG P1 用例平均分必须 >= 4.0。
- 普通聊天 P1 用例平均分必须 >= 4.0。
- Plan Agent 允许标记 beta，但不能影响普通聊天与 RAG 主流程。
- 不允许出现跨用户私有知识泄漏。
- 不允许出现长时间无响应且前端没有失败提示。

## 当前产品定位建议

v1 应定位为“企业私有知识库助手 + 科研学习辅助”，不要把 Plan-and-Execute 作为客户承诺的核心卖点。Plan 模式可以作为 beta 能力展示，但上线门禁不应依赖它完全稳定。
