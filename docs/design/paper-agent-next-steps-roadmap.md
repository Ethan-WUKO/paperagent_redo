# Paper Agent 下一步路线

更新时间：2026-07-07

## 当前共识

我们接下来不应该先做一个大的 Project 外壳，而是先把底层能力打稳：论文润色、文献推荐、评测体系、任务资产管理。Project 能力应该建立在这些稳定能力之上。

## 主线一：文献推荐能力独立化

文献重排序属于“文献推荐/文献检索”能力本身，不应该只写在论文润色流程里。

目标结构：

- 文献推荐模块负责 query 生成、OpenAlex/arXiv 检索、去重、规则打分、LLM rerank、推荐理由和 BibTeX 输出。
- 论文润色模块负责解析论文、识别 citation slot，然后调用文献推荐模块。
- 聊天 Agent 后续遇到“帮我找论文”也复用同一套文献推荐能力。
- Project 后续可以把这套能力作为项目文献库的入口。

优先要做：

- 明确文献推荐输入输出协议。
- 修正并验证当前规则排序、LLM rerank、类别平衡和去重逻辑。
- 输出可解释排序信息：规则分、LLM 选择理由、最终 selected 原因。
- 区分“已存在参考文献”和“真正新增推荐文献”。

## 主线二：论文润色质量闭环

论文润色要从“能跑完”升级到“可验证、可解释、可复现”。

优先要做：

- 固定样例集，保留输入论文、期望检查项和输出结果。
- 自动生成润色前后 diff。
- 检查每个章节是否真的改动、是否失败保留原文。
- 记录失败章节原因，例如模型失败、格式解析失败、保护规则触发。
- 加 LaTeX 编译验证，缺少模板文件时给出明确提示。
- 文献推荐结果要能回到 citation slot，看每条推荐支持哪个论点。

## 主线三：图表理解与图表建议

图表能力先做“理解与建议”，不要一开始就做自动改图。

优先要做：

- 解析 LaTeX 中的 figure/table、caption、label、正文引用。
- 抽取图片文件和表格内容。
- 让模型解释图表表达的实验结论。
- 检查正文是否准确描述图表。
- 生成图片修改建议，例如标注不清、曲线说明不足、caption 太弱。
- 后续再考虑图像编辑或自动生成替代图。

## 主线四：论文润色页面重做

当前页面功能堆叠明显，需要重新做信息架构。

建议布局：

- 左侧：任务列表，支持多个任务并行、恢复、筛选。
- 中间：当前任务流程，包括上传、解析、结构确认、文献推荐、润色、导出。
- 右侧：当前步骤结果，例如结构确认、推荐文献、diff、图表建议、下载。
- 顶部：任务状态、耗时、模型、成本、失败提示。

重点是让用户一眼知道：现在跑到哪一步、需要我做什么、结果好不好。

## 主线五：评测系统

评测系统必须尽早做，否则后续每次改 prompt、排序、模型，都不知道是变好还是变坏。

优先指标：

- 文献推荐召回质量。
- 文献推荐相关性。
- 新文献去重准确性。
- 润色章节成功率。
- 润色前后差异质量。
- LaTeX 编译成功率。
- 任务耗时和模型调用成本。

## 主线六：通用 Agent 能力

通用 Agent 可以继续扩，但要等工具注册、权限控制、工具筛选、过程展示更稳定后再大规模加。

后续方向：

- 编程能力。
- GitHub MCP。
- 文件读写和项目分析。
- 工具调用过程可视化。
- 权限与确认机制。
- 工具选择策略，避免一次性把所有工具都给模型。

## 主线七：Project 能力

Project 是上层容器，不是短期先做的第一步。

Project 后续应该统一组织：

- 论文任务。
- 文献库。
- 知识库。
- 聊天记录。
- Agent 任务。
- 代码仓库。
- 产物和版本。

Project 做之前，至少要先稳定论文润色、文献推荐、任务资产和评测系统。

## 推荐推进顺序

1. 先做文献推荐能力独立化和重排序评测。
2. 再做论文润色 diff、失败原因和编译验证。
3. 然后重做论文润色 UI。
4. 再加入图表理解。
5. 最后扩通用 Agent 和 Project。
