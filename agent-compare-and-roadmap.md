# Agent 对比分析与后续改造建议

## 文档目的

这份文档面向后续参与开发的 AI 开发者、架构设计者和任务排期负责人。

目标不是做代码级抄写，而是回答四个更重要的问题：

1. `paicli-main` 作为一个完整 agent 项目，用户给出任务后，系统是如何处理的？
2. 这个处理流程里，是否存在证据校验、结果校验、失败修复等环节？
3. `private-helper-test(0708)` 目前已经具备哪些 agent 能力，缺的又是什么？
4. 如果要以后者为主线持续建设，后续应该如何排工作、先改什么、后改什么？

---

## 一、结论先行

一句话概括：

- `paicli-main` 的优势是“像一个完整可用的 agent 产品”，任务入口清楚、执行路径自然、模式切换明确、用户体验顺。
- `private-helper-test(0708)` 的优势是“治理和可信度更强”，在 Project 约束、工具权限、证据有效性、完成校验、步骤校验这几件事上明显更严。

真正的问题不是：

- 前者强，后者弱

而是：

- 前者强在“完整 agent 外壳”
- 后者强在“受控 runtime 和证据闭环”

所以后续正确方向不是简单照搬 `paicli-main`，而是：

**借鉴 `paicli-main` 的统一任务流和产品化外壳，同时保留并强化 `private-helper-test(0708)` 已有的证据治理能力。**

如果做对了，最终产物应该是：

- 比 `paicli-main` 更可信
- 比当前 `private-helper-test(0708)` 更像真正可用的 agent

---

## 二、`paicli-main` 的典型任务处理流程

可以把 `paicli-main` 理解成一个“已经成型的 agent CLI 产品”。

当用户给它一个任务时，系统整体上会这样处理。

### 1. 先判断任务走哪条模式

它不是只有一种执行方式，而是有三条清晰路径：

- 普通任务：默认 ReAct
- 复杂任务：Plan-and-Execute
- 协作任务：Multi-Agent

从产品体验上，这一点非常重要。用户不会感觉自己在使用很多松散模块，而是感觉“我在跟一个 agent 交互，只是它内部会切换工作模式”。

这就是 `paicli-main` 最值得借鉴的第一点：

**模式很多，但入口统一，用户感知简单。**

### 2. 普通任务走 ReAct 循环

默认情况下，用户输入会进入一个典型的 ReAct 循环：

1. 记录用户输入
2. 注入必要上下文
3. 调用模型
4. 如果模型要求用工具，就执行工具
5. 把工具结果喂回模型
6. 模型继续思考
7. 直到不再请求工具，输出最终结果

这种模式的优点是：

- 对简单问题响应快
- 对局部任务足够灵活
- 不需要为所有任务都支付计划成本

也就是说，`paicli-main` 非常强调一件事：

**不是所有任务都要先做计划。**

### 3. 复杂任务走 Plan-and-Execute

如果用户显式走计划模式，或者系统认为任务更复杂，就会先做计划，再执行。

大致流程是：

1. 生成计划
2. 计划拆成多个任务节点
3. 计算依赖关系
4. 找出当前可执行的任务
5. 串行或并行执行
6. 某个任务失败时，根据进度判断是否重规划
7. 全部完成后汇总输出

它的特点不是“计划特别严”，而是“计划可执行、可重跑、可重规划”。

从产品侧看，这会给人一种成熟感：

- 能拆任务
- 能推进
- 能失败重来

### 4. 有安全控制和预算控制

`paicli-main` 不是完全放飞的。

它内部也有一些重要约束：

- 路径保护
- 命令保护
- 工具调用预算
- 上下文预算
- 会话压缩
- 快照恢复

这说明它不是“瞎跑型 agent”，而是“能跑、也知道收着点跑”的 agent。

### 5. 它有没有证据校验？

有“校验”，但主要是偏执行层和安全层，不是严格的证据可信性校验。

更准确地说，它的校验集中在这些方面：

- 计划失败了没有
- 任务执行报错了没有
- 是否超预算
- 是否要重规划
- 是否越权访问路径或危险命令

但是它没有看到特别强的、类似下面这种闭环：

- 这个结论是否只能由“本轮、当前、同项目、同版本”的证据支撑
- 这个文件观察是不是仍然和当前 manifest/hash 对得上
- 历史 tool 输出是否会被误当成当前可信证据
- 最终回答是否一定经过统一的 evidence gate 才能对外发布

所以可以把 `paicli-main` 的证据能力理解成：

- 有执行控制
- 有安全控制
- 有失败控制
- 但“证据可信性约束”不算特别重

## 三、`private-helper-test(0708)` 当前的典型处理流程

这个项目的现状和 `paicli-main` 非常不同。

它不像一个已经彻底打磨好的通用 agent 产品，更像一个：

**已经具备强治理、强约束、强审计能力的受控 agent runtime 内核。**

### 1. 请求先进入协调层，而不是直接扔给模型

这一步很关键。

它不是用户消息一来就“让模型先说”，而是先做一轮服务器侧判断：

- 有没有合法上下文
- 有没有 project 绑定
- 这个请求能不能走当前能力
- 该不该进入计划模式
- budget、provider、model、tool policy 是否已解析完成

这说明它的总体设计是：

**先做受控决策，再进入模型执行。**

这比很多普通 agent 系统更稳。

### 2. 它已经有统一 runtime 思路

虽然产品体验上还不够顺，但底层已经有统一 runtime 思路了：

- 策略选择
- runtime adapter
- completion verifier
- repair / reflection
- project runtime

这意味着：

它并不是“还没开始做 agent”，而是“已经做到了 agent runtime 的后半段”，只是前半段的产品化闭环还不完整。

### 3. 普通运行不是随便给结果，而是先过 completion gate

这点是它相对 `paicli-main` 最大的优势之一。

流程不是：

- 模型返回什么就展示什么

而是：

1. 运行器先得到原始结果
2. 再提取证据
3. 再判断结果是否足够完整
4. 再判断是否证据不足
5. 必要时再 repair 一次
6. 最后才生成对外结果

这说明这个系统天然更适合做：

- 需要可审计结论的 agent
- 需要受控项目读写边界的 agent
- 需要证据追溯的研究/工程 agent

### 4. Project 场景下有强绑定和强约束

这一点非常重要。

在这个系统里，Project 不是一个前端随便传个 id 就能信的概念。

它的设计思路是：

- 只有认证过的 Project runtime 入口，才能把 Project 上下文绑定进运行时
- 工具执行时会再次确认这个 Project 仍然有效
- 输出证据时必须带上可验证的项目信息
- 最终返回给前端时，还会再投影一遍安全 evidence

这说明它不是“把项目路径告诉模型”，而是“服务器替模型守住项目边界”。

这件事很值钱。

因为在真正做可落地 agent 产品时，最难的一直不是“让模型看起来聪明”，而是：

- 怎么证明它没越权
- 怎么证明它说的话是当前项目内观察出来的
- 怎么证明它没把旧结果当成新证据

`private-helper-test(0708)` 在这方面已经走得很前了。

### 5. Plan 执行已经比表面上强很多

它不只是“有个 planner”。

实际上，它已经有比较完整的 plan executor 特征：

- 持久化计划
- 步骤依赖
- 异步执行
- 并行步骤
- 步骤重试
- repair step
- degrade
- 事件记录
- step verifier
- project evidence gating

也就是说，它的 Plan 能力其实比很多表面上“会拆步骤”的 agent 更工程化。

## 四、`private-helper-test(0708)` 是否存在证据校验环节？

结论很明确：

**存在，而且很强。**

不仅存在，而且这部分能力明显强于 `paicli-main`。

### 1. 工具结果不是天然可信

Project 工具执行时，不是拿到输出就算完。

它会做两件事：

- 先确认调用时真的有受信 Project 上下文
- 再把结果包装成带项目信息、路径、版本、证据引用的结构

也就是说，工具输出不是裸字符串，而是带 provenance 的结构化结果。

### 2. completion verifier 会重新抽取当前可用证据

这个 verifier 的关键价值在于：

- 它不是只看模型说得像不像
- 它会看这次运行中到底观察到了什么
- 它会从 tool 消息中抽取当前 project evidence
- 它还会合并可信 ledger
- 然后再判断这个回答能不能成立

这是非常典型的“证据门”。

### 3. 证据必须经过当前 manifest 校验

这一步说明系统不是只做“有证据引用”这么浅的一层。

它会进一步验证：

- 这个证据是否属于当前 Project
- 这个文件路径是不是当前仍然可见
- 这个 hash/version 是否和当前 manifest 匹配

也就是说，它不信“历史上我见过一次这个文件”，它只信：

**当前这次、当前这个项目、当前这个版本下仍然成立的观察。**

这就是很硬的证据闭环。

### 4. plan step 还有单独的证据充足性检查

在步骤执行里，如果一个 step 声称完成了，但没有拿到当前授权文件的观察结果，系统会认为证据不足。

这说明它不是等最后统一失败，而是在 step 层就开始卡证据。

这一步的价值非常高，因为它能尽早阻止“看起来完成、实际上没证据”的假成功。

### 5. step verifier 不只是看“有没有输出”

它还会根据 success criteria 去判断：

- 该步是否真的满足目标
- 哪些 criterion 被满足
- 哪些没有满足
- 若返回格式有问题，是否允许做一次 bounded repair

这已经是“步骤完成性验证”，不是简单字符串判断。

### 6. 最终对外只释放 canonical answer

Project streaming 的设计也说明了这一点：

- 中间过程可以作为 process event 流出
- 但最终面向用户的 assistant answer 要等 completion gate 选定

这个设计很成熟。

因为很多 agent 系统的问题就是：

- 中间思考和最终回答混在一起
- 用户看到了未验证内容
- 后续 repair 结果又叠上来，导致答案污染

这个项目已经在避免这些坑。

## 五、两边真正的核心差异

到这里，核心差异其实已经很清楚了。

### 1. `paicli-main` 更像“完整产品壳”

它强在：

- 用户入口清晰
- 三种模式自然
- 执行路径简单直观
- 默认就是一个“能干活的 agent”

它解决的是：

**怎么让 agent 顺畅地跑起来。**

### 2. `private-helper-test(0708)` 更像“强治理内核”

它强在：

- Project 受控上下文
- runtime 协调
- tool allowlist
- completion verification
- evidence ledger
- manifest revalidation
- step verification
- repair/degrade

它解决的是：

**怎么让 agent 说的结果更可信、边界更清楚、运行更可审计。**

### 3. 前者偏“能跑”，后者偏“可信”

所以如果只看产品体验：

- `paicli-main` 更完整

如果只看工程可信性：

- `private-helper-test(0708)` 更严

### 4. 后者不是没有 agent，而是 agent 闭环还没有产品化

这是最重要的判断。

目前 `private-helper-test(0708)` 的问题不是：

- 缺 planner
- 缺 verifier
- 缺工具
- 缺 project context

而是：

**这些能力已经存在，但用户任务进入系统后，还没有被组织成一个足够顺、足够统一、默认可走通的 agent 产品闭环。**

## 六、`private-helper-test(0708)` 当前最主要的短板

下面这些点，才是后续工作安排最应该盯住的地方。

### 1. 缺少像 `paicli-main` 那样自然的统一入口感

现在它已经有 runtime、plan、project runtime、coordinator，但从产品观感上还不像“一个统一 agent”。

开发者能看懂结构，不代表用户会觉得自然。

真正成熟的效果应该是：

- 不管是普通问答、项目问答、复杂任务、计划执行
- 用户都觉得自己在跟一个系统交互
- 而不是在碰多套接口、多种运行器

### 2. 自动策略选择还不够强

现在的策略选择更像：

- 没工具就 DIRECT
- 有工具就 SINGLE_STEP_REACT
- 明确 trusted plan capability 才 PLAN_EXECUTE

这还不够像一个成熟 agent。

成熟 agent 更应该做到：

- 简单问题直接答
- 稍复杂的问题单步 ReAct
- 多步复杂目标自动进入 plan

也就是：

**计划不是显式高级功能，而是系统自然会用的能力。**

### 3. 普通 chat 和 plan 的产品抽象还不统一

从底层看，它们已经在靠近统一。

但从开发组织和后续扩展看，还缺一个足够清楚的抽象：

- 一次用户任务到底是什么
- 一次 run 的边界是什么
- run 的 trace/event/tool/evidence 怎么统一管理
- 普通回答和计划回答最后怎么落成同一种 canonical outcome

这个抽象不立起来，后面越做越容易碎。

### 4. 强证据能力目前更多存在于“约束层”，还没充分转化成“产品能力”

现在系统内部很清楚：

- 有 trusted evidence
- 有 current manifest revalidation
- 有 verifier

但如果产品层和调度层没有把这些能力组织好，最终就会出现一个现象：

- 内核很强
- 对外体验一般

这会让团队误判成“是不是模型不够聪明”，其实不是。

问题往往出在：

- 任务如何被分流
- 证据门如何进入主流程
- 最终结果如何被组织输出

### 5. 单 agent 主链还没真正“收口”

目前最应该优先做的是把单 agent 主链收口，而不是先扩多 agent。

如果单 agent 的路径还不够统一，那么多 agent 只会让问题变复杂：

- trace 更乱
- evidence 更难追
- repair 更难管
- 策略更难选

所以在排期上，应该明确：

**先把单 agent 主链做好，再做多 agent 扩展。**

---

## 七、后续建议的总体方向

建议不要把目标定成：

- “做成另一个 paicli-main”

而应该定成：

- “做一个比 `paicli-main` 更可信、但同样顺手的受控 agent”

这个目标更符合当前基础。

因为 `private-helper-test(0708)` 已经有两样特别值钱的东西：

1. Project 受控上下文
2. 证据可信性闭环

这两样如果做成产品级能力，会非常有竞争力。

---

## 八、建议的改造方案

下面给的是适合拿来排后续工作的版本。

不是一次性大重构，而是分阶段推进。

---

## 阶段一：先把“统一任务入口”做起来

### 目标

先解决“系统看起来像很多块拼起来的”问题。

### 这一阶段要做什么

1. 定义统一的单次任务运行主线

建议明确一条统一链路：

`用户请求 -> 请求解析 -> 策略选择 -> 上下文构建 -> runtime 执行 -> completion 校验 -> canonical 输出 -> trace 持久化`

不管最终是 DIRECT、REACT 还是 PLAN，都必须走同一条主线。

2. 明确“一次 run”的产品抽象

要让团队对这些概念有统一认知：

- 什么是一轮用户任务
- 什么是过程事件
- 什么是最终答案
- 什么是中间尝试
- 什么是可对外证据

3. 把 Project chat、普通 chat、plan create/execute 都纳入统一调度视角

不是说内部不能分 adapter，而是要让系统入口层和任务层有统一语义。

### 这一阶段解决的问题

- 系统形态不统一
- 后续功能越做越碎
- 不同路径各自长逻辑，难以持续维护

### 这一阶段不该做什么

- 不要先扩多 agent
- 不要先加更多花哨工具
- 不要先改大模型提示词做表层修补

---

## 阶段二：补“自动策略选择”，让系统更像真正 agent

### 目标

让系统自己决定什么时候直接答、什么时候单步 ReAct、什么时候进入计划。

### 这一阶段要做什么

1. 升级策略选择器

从现在这种偏保守的“有无工具”判断，升级成复杂度感知判断。

至少要考虑这些维度：

- 请求是否明显是简单问答
- 是否需要多次外部/项目观察
- 是否需要多步依赖
- 是否需要明确验证完成条件
- 是否是用户显式要求分步骤完成

2. 给普通请求自动进入计划的能力

现在计划能力更像“显式走 API 的高级功能”。

应改成：

- 简单问答不用计划
- 多步研究/分析任务自动计划
- 带复杂验证目标的任务优先计划

3. 让策略切换可追踪

每次选择 DIRECT / REACT / PLAN，都要有清楚原因。

不是为了日志好看，而是为了后续排障、调优和评估。

### 这一阶段解决的问题

- 用户复杂任务没有自然进入 plan
- 系统像聊天机器人而不像任务 agent
- 计划能力存在但利用率不自然

### 风险提醒

这一阶段最容易犯的错是：

- 一看复杂就全进 plan

这会把系统做笨重。

正确做法是：

- 让 plan 成为“必要时自动使用的能力”
- 而不是“默认所有任务都先计划”

---

## 阶段三：把现有证据治理能力正式产品化

### 目标

把现在已经存在的 evidence / verifier / project gating，从“底层强约束”升级成“主流程中的一等能力”。

### 这一阶段要做什么

1. 把 evidence gate 明确成统一阶段

无论是普通回答、Project 回答、Plan step 结果，最后都要经过统一 completion gate。

现在已经有这个方向，但要更清楚、更稳定、更统一。

2. 统一对“证据不足”的系统表现

证据不足时，系统应该有固定行为：

- 返回不足
- 说明为什么不足
- 尝试有限 repair
- 必要时进入 partial / degraded，而不是假装成功

3. 统一对用户展示“过程”和“最终结果”的边界

过程事件可以保留，但最终对用户只发布 canonical answer。

不要让未验证尝试、旧轮结果、修复前的中间话术污染最终回答。

4. 把可信证据链真正融入后续产品能力

比如：

- 建议变更
- 待审候选修改
- 可审计结论
- 可引用 project evidence

这些能力的前提，其实就是现在这套 evidence 体系。

### 这一阶段解决的问题

- 内核很强但外部感知不到
- 最终结果的可信性规则分散
- 后续做读写型 agent 时没有统一可信基座

---

## 阶段四：在主链稳定后，再考虑多 agent 和并行子任务

### 目标

在单 agent 真的稳定以后，再做更复杂的协作增强。

### 为什么要放后面

因为多 agent 会放大所有未解决问题：

- trace 管理更难
- 证据归属更难
- repair 路径更复杂
- tool policy 更容易出错

所以正确顺序是：

1. 单 agent 主链统一
2. 自动策略切换稳定
3. completion/evidence gate 稳定
4. 再考虑子任务并行和多 agent 协作

### 适合后续再做的方向

- read-heavy 并行研究子任务
- 计划内子步骤并行扩展
- reviewer / planner / worker 分工
- 更强的自动 reflection

---

## 九、给开发排期的具体建议

如果后续要转成工作安排，我建议按下面方式拆。

### P0：必须先做

1. 统一单次任务主流程抽象
2. 统一普通 chat / Project chat / Plan 的入口视角
3. 补 run trace 的统一语义
4. 保证最终只输出 canonical answer

这是地基。

不先做这个，后面所有“增强能力”都会继续长在分裂结构上。

### P1：接着做

1. 升级策略选择器
2. 引入复杂度驱动的自动 plan
3. 让 plan 成为系统自然能力，而不是特殊路径

这是把系统真正做成 agent 的关键。

### P1：与上面并行推进

1. 统一 completion gate 规则
2. 统一 insufficient evidence / partial / degraded 语义
3. 统一 step verifier 和 final verifier 的产出语义

这是把系统真正做成“可信 agent”的关键。

### P2：等主链稳后再做

1. 多 agent / 子任务并行
2. 更复杂的 reflection
3. 更丰富的研究型工具组合
4. 更强的历史/记忆增强

这类工作价值高，但不该抢在主链收口之前。

## 十、对 AI 开发者的直接建议

如果后续是交给 AI 开发者持续推进，建议注意以下原则。

### 1. 不要把这件事理解成“补几个工具”

当前短板不是工具数量，而是任务流闭环。

### 2. 不要把这件事理解成“继续加强证据规则”

证据规则已经不弱了。

当前短板不是“证据不够严”，而是“严的能力还没完全变成顺的产品”。

### 3. 不要把这件事理解成“直接照抄 paicli-main”

应借鉴的是：

- 统一入口
- 统一模式抽象
- 自然的任务处理流

不应照抄的是：

- 把证据要求做轻
- 把 Project 约束做弱
- 让模型自由覆盖后端治理

### 4. 先做收口，再做扩展

最容易失败的路线就是：

- 主链没收口
- 就开始加反思、多 agent、更多工具、更多模式

这样会很快失控。

### 5. 把“可信”和“顺手”同时作为目标

这不是二选一。

当前项目最有价值的方向，就是：

- 保持可信
- 提高顺手

如果只提升顺手，容易退化成普通 chat+tools。

如果只提升可信，容易继续停留在“强内核、弱产品”的状态。

---

## 十一、最终建议

最终建议可以概括成三句话：

1. `private-helper-test(0708)` 不需要从零重做 agent，它已经有很强的 runtime、plan、evidence、verifier 基础。
2. 真正要补的是“统一任务主链、自动策略选择、产品化闭环”，也就是把这些强能力装成一台完整整机。
3. 后续排期应优先做主链收口和策略升级，暂缓多 agent 扩展；并始终保留现有 Project 证据治理作为系统核心优势。

如果这个方向执行到位，最终结果不应只是“像 `paicli-main` 一样能跑”，而应是：

**一个比 `paicli-main` 更可信、比当前 `private-helper-test(0708)` 更完整的受控 agent 系统。**
